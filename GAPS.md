# ArcLink Gaps

> AUDIT TARGET — generated by the Ralphie journey/gaps audit
> (`./ralphie.sh --arclink-user-journey-audit`). This file is intentionally a
> stub at launch. Every row below must be DERIVED FROM FRESH SOURCE EVIDENCE.

## Audit mandate

Compare the full `USER_JOURNEY.md` story against the actual repository — code,
tests, docs, service units, configuration, scripts, web, plugins, and
control-node / shared-host boundaries — and register every missing, partial,
proof-gated, policy-gated, weakly-tested, confusing, or risky joint.

CRITICAL — disprove, do not confirm: `research/PRODUCT_REALITY_MATRIX.md`
reports counts such as "0 gap, 0 partial." Treat those as an UNVERIFIED CLAIM TO
DISPROVE, not a starting truth. Do not copy matrix rows. Independently
re-confirm or knock down each claim against source, and actively hunt for
net-new gaps the matrix missed. A finding of "the matrix says real but local
evidence is thin/absent" is itself a gap worth recording.

The seed at `research/seed-gaps-draft.md` is input only — its rows are a prior
draft, not a baseline to preserve.

## Taxonomy

| Status | Meaning |
| --- | --- |
| `gap` | The intended behavior is absent or contradicted by source evidence. |
| `partial` | Some implementation exists, but the end-to-end journey is incomplete. |
| `proof-gated` | Local code exists or intent is clear, but live/external proof is required. |
| `policy-question` | Product, security, pricing, or operations choice cannot be decided by code. |
| `test-gap` | Code may exist, but local regression proof is missing or too weak. |
| `doc-gap` | User/operator documentation is absent, misleading, or stale. |
| `ux-gap` | Behavior exists but the user-facing surface is confusing or too hidden. |
| `ops-gap` | Operational path is unclear, manual, or missing safe runbook coverage. |
| `security-risk` | A path could violate isolation, secret safety, payment trust, or authority boundaries. |
| `real` | Source evidence and tests support the claim locally (cite both). |

| Severity | Meaning |
| --- | --- |
| P0 | Blocks trust, user isolation, payment, provisioning, data safety, or secret safety. |
| P1 | Blocks a core Captain journey or operator journey. |
| P2 | Causes degraded behavior, confusing recovery, or incomplete confidence. |
| P3 | Polish, scale hardening, future product option, or documentation depth. |

## Required sections (to be filled by the audit)

- Gap register: one row per finding with id, severity, status, journey joint,
  expected behavior, actual evidence (with source paths/lines), missing
  proof/tests, user impact, likely owner/surface, and recommended next repair.
- Not Gaps / Already Real: surfaces checked and found genuinely covered, each
  with the source + test evidence that proves it.
- Proof Gates: exact live credentials or authorized proof runs required before
  any `proof-gated` row can move to `real`.
- Policy Questions: choices code cannot decide.
- Test Plan: focused local, no-secret checks for every code-owned gap.
