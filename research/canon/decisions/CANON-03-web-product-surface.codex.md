## CANON-03 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Admin CIDR allowlist must not depend on opaque Next rewrite XFF behavior
- RECOMMENDATION: Make `control-ingress`/Traefik the production API transport owner for `/api`, remove or dev-gate the unconditional `web/next.config.ts` rewrite, and fail closed in `python/arclink_hosted_api.py` when CIDR-protected routes arrive from loopback/trusted proxy without a real forwarded client IP. Split proxy trust from operator allowlist with explicit config such as `ARCLINK_TRUSTED_PROXY_CIDRS` and `ARCLINK_ADMIN_ALLOWED_CIDRS`, preserving `ARCLINK_BACKEND_ALLOWED_CIDRS` only as a migration alias. Add tests proving: no XFF from loopback to admin routes is 403, allowed forwarded operator IP is 200, spoofed XFF from an untrusted peer is 403, and `/auth/login` cannot mint admin sessions without the same resolved client-IP policy. Add named live proof `PG-ADMIN-CIDR-XFF` under ingress proof.
- SYMPHONY ANCHOR: North Star — “Operators own the universe: hosts, secrets, fleet, policy…”; Whole-System Traversal — “how it fails closed”.
- RATIONALE: Current code trusts forwarded headers only after direct-peer trust, but loopback is always backend-allowed, so a missing XFF through the Next rewrite can collapse admin CIDR to `127.0.0.1` and pass. Traefik already routes `/api` directly to `control-api`; make that the source-owned transport contract instead of relying on Next standalone behavior. This keeps operator network policy on the operator-owned ingress/API layer, not in Captain-facing page code, and gives dashboard, API, docs, and tests the same truth.
- TRADEOFFS & ALTERNATIVES: Merely proving Next forwards XFF in one live deployment is weaker because it leaves a release/runtime-dependent implicit contract. Edge-only allowlisting is not enough because the hosted API would still have a local fail-open path. The residual risk is migration friction for existing installs with ambiguous `ARCLINK_BACKEND_ALLOWED_CIDRS`; handle that by preserving config, warning clearly, and requiring reconfigure before claiming `PG-ADMIN-CIDR-XFF`.
- EFFORT / BLAST-RADIUS: high — touches hosted API IP resolution/auth, Docker/Traefik ingress config, Next config or dev proxy behavior, deploy config generation, API docs, tests, and live proof.

### DECISION 2: Dynamic ArcPod/fleet dashboard URLs need server-side host policy, not arbitrary `https://`
- RECOMMENDATION: Keep the client `http`/`https` scheme guard as a last-mile UI safety net, but add a server-owned access URL policy before any ArcPod/fleet dashboard URL is persisted or rendered. Implement a shared validator around `arclink_access_urls(...)` that allows only generated domain-mode hosts, generated Tailscale path/port hosts, active worker private-mesh hosts, and explicit operator-configured custom domains/host suffixes from private config. Apply it at the write/read boundaries now accepting stored `metadata.access_urls`: `python/arclink_sovereign_worker.py`, `python/arclink_provisioning.py`, `python/arclink_dashboard.py`, `python/arclink_public_bots.py`, and hosted API payloads via those helpers. Invalid URLs should be withheld from Captain/Raven/web surfaces, preserved in private metadata with a redacted policy-block event, and surfaced as “dashboard link blocked by URL policy” with an operator repair action. Add local regressions for domain, custom-domain, Tailscale path, Tailscale port, WireGuard/private-mesh, and hostile `https://evil.example` metadata; add live proof `PG-INGRESS-ACCESS-URLS`.
- SYMPHONY ANCHOR: Third-Party Integration Boundaries — “Cloudflare and Tailscale own ingress primitives; ArcLink owns desired-state records…”; North Star — “Captains own their Pods and Crew, not the host.”
- RATIONALE: The current repair blocks non-http schemes, but stored `access_urls` are still accepted if they merely start with `https://` in several server surfaces. Host policy is an operator/deployment decision because legitimate links vary by wildcard domain, Tailscale, worker DNS, and WireGuard/private mesh. Central validation makes Raven, dashboard, hosted API, and worker recovery show the same system truth while failing closed without deleting Pod state.
- TRADEOFFS & ALTERNATIVES: Hard-coding one public suffix would be simpler but would break custom domains and tailnet/private-mesh deployments. Leaving it client-only would not protect Telegram/Discord handoff buttons or hosted API consumers. The residual risk is over-broad operator suffix config; mitigate with preview/dry-run validation and redacted evidence before enabling custom host policy.
- EFFORT / BLAST-RADIUS: high — touches shared URL generation/policy, provisioning intent, worker metadata persistence, hosted API/dashboard/public bot rendering, docs, local tests, and ingress live proof.
