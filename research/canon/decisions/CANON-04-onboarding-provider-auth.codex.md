## CANON-04 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: NEW onboarding should have a real terminal `completed` state
- RECOMMENDATION: Make `completed` the canonical terminal success status for NEW public onboarding. Keep `first_agent_contact` as the event/current_step, but stop writing `status="first_contacted"` from `record_arclink_onboarding_first_agent_contact` in [python/arclink_onboarding.py](/root/arclink/python/arclink_onboarding.py:887). On first contact, require a paid/provisioning-ready linked deployment, write `status="completed"`, set `completed_at`, record the existing `first_agent_contact` event, and leave deployment status unchanged. Add a migration that backfills old `first_contacted` session rows to `completed`, preserves user/deployment/metadata, and drops/recreates `idx_arclink_onboarding_active_identity` without `first_contacted`.
- SYMPHONY ANCHOR: “Configuration, Schema, And Migration” — “Database schema changes are migration-aware, idempotent, reversible where practical, and tested against old-state fixtures.”
- RATIONALE: `completed` is already declared in the schema [python/arclink_control.py](/root/arclink/python/arclink_control.py:1385), and docs say completed rows are terminal, but current code’s active index still treats `first_contacted` as active [python/arclink_control.py](/root/arclink/python/arclink_control.py:2310). Moving success to `completed` gives every surface one system truth, frees channel identity for explicit later onboarding, and fails closed on invalid terminal resurrection.
- TRADEOFFS & ALTERNATIVES: Reject leaving `first_contacted` forever-active; it preserves today’s behavior but keeps a dead public contract. Reject deleting `first_contacted` outright; old rows, dashboard booleans, and bot routing need a compatibility window. Residual risk is public-bot post-launch command behavior, so cover `_latest_session_for_contact`, dashboard bot-contact, worker notification, and old-state migration tests.
- EFFORT / BLAST-RADIUS: med + schema/index migration, CANON-04 state machine, worker callback expectations, dashboard/public-bot/web status tests, docs/OpenAPI contract.

### DECISION 2: OLD completion must stop putting dashboard passwords in chat
- RECOMMENDATION: Replace the scrub-on-ack plaintext password message in [python/arclink_onboarding_completion.py](/root/arclink/python/arclink_onboarding_completion.py:307) with a credential-handoff flow. Do not include `access["password"]` in `full_text`; instead create a revealable handoff record, private-channel bound to the onboarding recipient, with TTL, `revealed_at`, `acknowledged_at`, and `removed_at`. Generalize `arclink_credential_handoffs` beyond deployment-only targets, or add equivalent subject fields, so OLD `linked_agent_id`/Hermes-home access can use the same reveal/ack/audit semantics as public Raven credentials.
- SYMPHONY ANCHOR: “Secrets, Keys, And Rotation” — “Public docs, chat transcripts, logs, evidence artifacts, command arguments, and generated markdown must never contain secret values.”
- RATIONALE: Current code prints the shared password into Discord/Telegram text before ack [python/arclink_onboarding_completion.py](/root/arclink/python/arclink_onboarding_completion.py:411). The newer Raven handoff rail already has the right shape: secret refs, private delivery checks, reveal-once behavior, ack/removal, and redacted audit/event evidence [python/arclink_public_bots.py](/root/arclink/python/arclink_public_bots.py:3549). This makes secrets operator-owned private state while Captains get a controlled handoff.
- TRADEOFFS & ALTERNATIVES: Reject keeping scrub-on-ack as the default; message edits are best-effort and chat platforms may retain history. Reject “DM only but still plaintext in first message”; private chat is not a secret store. Residual risk is recovery friction, so add a reissue/rotate path rather than reopening removed handoffs.
- EFFORT / BLAST-RADIUS: high + OLD curator completion, Telegram/Discord callbacks, credential-handoff schema/helper, dashboard password secret-source normalization, completion message tests, bot live proof under `PG-BOTS`.

### DECISION 3: Checkout should reserve locally, but materialize deployments only after paid Stripe proof
- RECOMMENDATION: Split “checkout intent/reservation” from `arclink_deployments`. Today `open_arclink_onboarding_checkout` calls `prepare_arclink_onboarding_deployment` before Stripe [python/arclink_onboarding.py](/root/arclink/python/arclink_onboarding.py:652), and the reservation helpers commit deployment rows immediately [python/arclink_control.py](/root/arclink/python/arclink_control.py:3647). Replace that with a durable onboarding checkout-intent/prefix-reservation row: commit the intent before Stripe, call Stripe with `onboarding_session_id` and `checkout_intent_id`, then on paid webhook atomically create the deployment rows, set entitlement, advance to `provisioning_ready`, and mark the intent materialized.
- SYMPHONY ANCHOR: “Billing, Entitlements, And Refuel” — “Entitlement state gates provisioning and provider continuation.”
- RATIONALE: This preserves crash recovery without creating orphan `entitlement_required` deployments when Stripe creation fails. CANON-04 owns the checkout intent, CANON-07 owns signed Stripe proof, and CANON-01/08 own deployment/provisioning materialization. Missing or conflicting intent data should fail closed, record redacted evidence, and not provision.
- TRADEOFFS & ALTERNATIVES: Reject simply wrapping current `prepare` in rollback; Stripe success plus local rollback creates ambiguity. Reject “Stripe first, no local row”; a crash after Stripe creation loses local ownership evidence. Residual risk is API compatibility because tests currently expect `deployment_id` during checkout; preserve a `planned_deployment_id`/compat field but label it as not materialized until webhook proof.
- EFFORT / BLAST-RADIUS: high + schema, onboarding checkout API, Stripe metadata/webhook sync, public bot direct checkout, web checkout status, fake E2E, live `PG-STRIPE`, and provisioning dry-run tests.
