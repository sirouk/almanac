## CANON-06 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Discord operator DM currently allows operator commands without explicit Discord user/role allowlist
- RECOMMENDATION: Change the default to fail closed. In `python/arclink_curator_discord_onboarding.py`, remove the `guild is None -> True` fallback in `_operator_discord_subject_allowed` and require `ARCLINK_OPERATOR_DISCORD_USER_IDS` for DM operator control, or user/role allowlists for guild control. Keep Discord DM as a valid operator channel, but make the speaker identity explicit in private config. Add install/reconfigure prompts and `arclink-ctl channel reconfigure operator` support to collect Discord operator user/role IDs; if a live Discord bot token is available, validate the configured channel and record redacted evidence under `PG-BOTS`. Update tests that currently assert DM-without-allowlist remains available, especially `tests/test_arclink_operator_raven.py`, to assert no-allowlist Discord DMs are rejected while allowlisted DMs pass.
- SYMPHONY ANCHOR: **Operator Raven / Admin Dashboard, API, And CLI Control**: “Only the configured operator identity/channel may execute operator actions.” Also **Identity, Access, And Session Governance**: “Rate limits, replay protection, nonce/confirmation, channel allowlists, and reason capture should be consistent enough that the same action cannot be made safer or more dangerous merely by choosing chat instead of dashboard.”
- RATIONALE: Current code lets `_operator_discord_subject_allowed` approve any non-bot sender in the configured Discord DM when no Discord allowlist exists (`python/arclink_curator_discord_onboarding.py:273-294`), and both slash/component and text operator paths trust that verdict (`:413-437`, `:586-593`, `:1128-1154`). That is weaker than the symphony’s “identity/channel” model because a Discord DM channel ID is not the same primitive as a Discord user ID. The local source owner should be private config: `ARCLINK_OPERATOR_DISCORD_CHANNEL_ID` plus `ARCLINK_OPERATOR_DISCORD_USER_IDS` / `_ROLE_IDS`. Existing outbound notifications should continue; only inbound operator mutation/control should block until policy is explicit. This preserves state, fails closed, and makes Telegram/Discord/dashboard safety posture converge.
- TRADEOFFS & ALTERNATIVES: I rejected leaving the implicit DM trust as-is because it depends on channel identity alone and conflicts with the Discord-specific symphony line that secondary Discord control is gated by user/role allowlist plus explicit channel. I also rejected banning Discord DM operator channels entirely; solo operators can still use a DM if their Discord user ID is allowlisted. Residual risk remains account compromise, which belongs to the existing approval-code/nonce/audit rails for dangerous actions.
- EFFORT / BLAST-RADIUS: med — touches the Discord curator adapter, deploy/bootstrap or `arclink-ctl` operator-channel configuration, operator health/status messaging, and focused regression tests. No DB schema or Captain onboarding state migration is required.
