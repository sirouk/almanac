## CANON-07 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Full forged-metadata policy when no Stripe customer or subscription is locally bound
- RECOMMENDATION: Make first-binding local-source-owned, not metadata-owned. Keep the current conflict guard in `_assert_entitlement_account` (`python/arclink_entitlements.py:433-479`, called at `:794-801`), but change the subscription branch resolver (`:734-743`) so unbound `customer.subscription.*` and `invoice.*` events cannot grant or move entitlement from `arclink_user_id` metadata alone. Only `checkout.session.completed` may create the first binding, and only when `obj.id` matches a local `arclink_onboarding_sessions.checkout_session_id` row whose `user_id`, `deployment_id`, `client_reference_id`, and onboarding session metadata agree. Existing local bindings from `arclink_subscriptions` or `arclink_users.stripe_customer_id` should remain authoritative; metadata mismatch should fail closed with redacted evidence, not silently reassign ownership. Move email merge (`python/arclink_entitlements.py:750-793`; repoints in `python/arclink_control.py:3848-3865`) behind that local-source-owner proof, because email merge is powerful state movement. Add tests for: unbound subscription metadata rejected; locally opened checkout first-binds; out-of-order subscription event fails/replays after checkout; customer/subscription local-owner mismatch fails; no `set_arclink_user_entitlement` metadata-only auto-created user path (`python/arclink_control.py:3924-3932`).
- SYMPHONY ANCHOR: `Third-Party Integration Boundaries`: “Stripe owns payment collection and subscription events; ArcLink owns entitlement interpretation, idempotency, gating, audit, and recovery copy.” Also `Billing, Entitlements, And Refuel`: “Entitlement state gates provisioning and provider continuation.”
- RATIONALE: This preserves the intended split: Stripe can say “a paid subscription event happened,” but ArcLink decides which Captain owns the entitlement using local checkout/session/binding state. It fails closed when the system cannot prove ownership, preserves existing user/subscription/deployment state by default, and leaves a replayable failed webhook row through the existing failed/received replay path (`python/arclink_entitlements.py:617-632`, `:910-919`). It also keeps every surface on the same truth: billing dashboards, Raven, provisioning gates, and recovery all read the Control Node DB instead of reinterpreting Stripe metadata differently. For legitimate imports or unusual Stripe-side repairs, add an operator-only typed recovery action with dry-run, reason capture, and redacted audit, rather than an env flag that lets metadata write entitlement.
- TRADEOFFS & ALTERNATIVES: Keeping the current wider first-binding contract is more compatible with arbitrary signed Stripe events, but it lets a Stripe metadata mistake or webhook-secret compromise target the wrong local user. Requiring every customer/subscription to be prebound before checkout is stricter but too brittle for normal self-serve checkout. Silently preferring local owner over mismatched metadata improves availability but hides a serious configuration conflict. The recommended path may temporarily 400/retry out-of-order subscription events until the checkout event binds the subscription; that is acceptable because it is replayable and state-preserving. Live Stripe ordering and payload shape should remain under `PG-STRIPE`.
- EFFORT / BLAST-RADIUS: med. Touches `python/arclink_entitlements.py`, focused entitlement/onboarding webhook tests, hosted API Stripe webhook fixtures that bypass real checkout state, and operator recovery/evidence docs if a manual bind action is added.
