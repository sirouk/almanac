## CANON-10 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Existing fleet hostname re-registration must not silently clobber capacity
- RECOMMENDATION: Make hostname re-registration idempotent, not mutating, unless it proves the same host identity or runs through an explicit operator replace/adopt action. Today `register_fleet_host` matches only `LOWER(hostname)` and updates `capacity_slots`/region/tags/metadata (`python/arclink_fleet.py:190-223`), while the schema enforces hostname uniqueness (`python/arclink_control.py:2541`). Change the contract so default re-register returns the existing row only when mutable fields are unchanged or same-host evidence matches; otherwise raise `ArcLinkFleetError`, audit `fleet_hostname_collision_blocked`, and leave capacity untouched. Allow updates only when `host_id` matches, linked inventory identity matches, or an operator-confirmed replace flow drains/removes/adopts with dry-run, reason, audit, and post-probe evidence.
- SYMPHONY ANCHOR: **Fleet, Provisioning, Ingress, And Recovery**: “Fleet hosts are registered with hostname, SSH endpoint, user, region, tags, capacity, state root, and health/probe evidence.” Also **North Star**: “Operators own the universe: hosts, secrets, fleet, policy...”
- RATIONALE: Hostname alone is not proof of host ownership. Failing closed on a collision preserves the old worker’s state and capacity truth, prevents over/under-scheduling, and keeps operator policy in the owner lane. The source owner is `python/arclink_fleet.py`; `python/arclink_inventory.py` and enrollment paths should pass identity evidence instead of relying on hostname.
- TRADEOFFS & ALTERNATIVES: Reject keeping current implicit mutation because reused DNS names can rewrite capacity with no proof. Reject always creating a new host because the DB intentionally treats hostname as unique and idempotent install/register flows need stable rows. Residual risk: legacy scripts/tests expecting capacity refresh by hostname must move to explicit same-host or replace semantics.
- EFFORT / BLAST-RADIUS: **med**. Touches `python/arclink_fleet.py`, inventory/enrollment registration call sites, deploy worker registration snippets, CLI/operator docs, and tests around fleet re-register, image-sync preservation, and hostname collisions.

### DECISION 2: Zero ASU is a valid blocking state, but zero observed hardware and unlinked load must not be authoritative
- RECOMMENDATION: Keep zero-capacity summaries as the fail-closed capacity state, but require positive observed RAM/disk before any probe path marks a machine `ready`. `compute_asu` currently returns `0` for nonnegative but insufficient RAM/disk (`python/arclink_asu.py:57-61`); preserve that for pending, not-yet-probed, or too-small workers. Add strict observed-hardware validation in ready writers such as `probe_inventory_machine` and `arclink_fleet_inventory_worker` so `ram_gib <= 0` or `disk_gib <= 0` degrades the machine instead of publishing ready zero capacity. For `current_load`, stop treating an empty `machine_host_link` as authoritative: it currently returns stored `asu_consumed` (`python/arclink_asu.py:72-74`) and `list_inventory_machines` silently falls back to stored load (`python/arclink_inventory.py:363-366`). Make unlinked load report `load_source=unlinked` and `asu_consumed=0` or raise internally; preserve any old nonzero value in metadata/audit during an idempotent migration. Placement remains governed by linked host truth: `list_fleet_hosts` reads linked inventory rows (`python/arclink_fleet.py:334-357`) and rejects `asu_available < 1` under `standard_unit` (`python/arclink_fleet.py:142-143`).
- SYMPHONY ANCHOR: **Fleet, Provisioning, Ingress, And Recovery**: “Placement rejects unhealthy, drained, or insufficient-capacity workers.” **Whole-System Traversal**: “Operator Raven, admin dashboard, CLI, diagnostics, live proof, and evidence rails show the same system truth.”
- RATIONALE: Zero capacity is useful and safe when it means “blocked/not enough capacity”; it is unsafe when it means malformed hardware proof. This keeps the DB/read-model contract state-preserving while preventing stale unlinked consumption from becoming product truth. Operators still own capacity policy, and all surfaces can show the same blocked/unlinked reason.
- TRADEOFFS & ALTERNATIVES: Reject making every zero ASU globally illegal because pending cloud-created machines and intentionally too-small workers need a non-destructive blocked state. Reject leaving stale unlinked `asu_consumed` because dashboard/API can display false capacity pressure even though placement uses linked hosts. Residual risk is mostly compatibility: old inventory rows need a migration/read-model quarantine path.
- EFFORT / BLAST-RADIUS: **med**. Touches `python/arclink_asu.py`, `python/arclink_inventory.py`, `python/arclink_fleet_inventory_worker.py`, dashboard/API inventory serialization, schema/migration evidence if preserving stale values, and focused ASU/inventory/fleet/dashboard tests.
