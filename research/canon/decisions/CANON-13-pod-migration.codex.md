## CANON-13 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Target-host-scoped health verification
- RECOMMENDATION: Add first-class scoped health evidence. `arclink_service_health` is currently keyed only by `(deployment_id, service_name)` in [python/arclink_control.py](/root/arclink/python/arclink_control.py:1300), while `_default_verifier` reads only `WHERE deployment_id = ?` in [python/arclink_pod_migration.py](/root/arclink/python/arclink_pod_migration.py:640). Add host/migration scope columns to the summary row and an append-only `arclink_service_health_checks` table keyed by `deployment_id`, `service_name`, `host_id`, `placement_id`, `migration_id`, `operation_idempotency_key`, `project_name`, `checked_at`. Add an executor health/compose-ps wire after `docker_compose_apply`, write target-host checks with `commit=False`, and make `_default_verifier` require fresh checks for `row["target_host_id"]` and `row["migration_id"]`. Missing, stale, wrong-host, or unhealthy rows must fail closed and rollback.
- SYMPHONY ANCHOR: Observability, SLOs, Capacity, And Scale: “Health is layered: ... fleet capacity, ArcPod health, workspace health, backup freshness, qmd/memory freshness, and proof status.” Configuration, Schema, And Migration: “Database schema changes are migration-aware, idempotent, reversible where practical, and tested against old-state fixtures.”
- RATIONALE: This makes the target host the local source owner of the proof instead of letting deployment-wide health rows stand in for reality. It preserves same-truth surfaces by letting dashboard/API/Raven show the same scoped health evidence the migration verifier used. It also keeps the current fail-closed behavior: no target-host proof means no promotion.
- TRADEOFFS & ALTERNATIVES: Reject detail-json-only host metadata because it is not enforceable and leaves the PK ambiguity. Reject accepting fresh deployment-wide rows because a source or unrelated writer can still satisfy the verifier. Residual risk is schema blast radius across provisioning, sovereign worker health writes, hosted API, dashboard, Operator Raven, and tests.
- EFFORT / BLAST-RADIUS: high; touches `arclink_control` schema/helpers, executor health wire, `arclink_pod_migration`, provisioning/worker health writers, API/dashboard/Raven health reads, schema and migration tests, plus live `PG-FLEET/PG-PROVISION` proof.

### DECISION 2: Dry-run-to-live reuse of one `migration_id`
- RECOMMENDATION: Allow same-`migration_id` dry-run-to-live only through explicit promotion, with phase-scoped idempotency keys. Current code uses one `arclink:migration:<migration_id>` key in [python/arclink_pod_migration.py](/root/arclink/python/arclink_pod_migration.py:1203), while `_operation_intent` includes `dry_run` in [python/arclink_pod_migration.py](/root/arclink/python/arclink_pod_migration.py:467), so the idempotency digest rejects live reuse after dry-run via [python/arclink_control.py](/root/arclink/python/arclink_control.py:3372). Use deterministic keys like `arclink:migration:<id>:dry-run` and `arclink:migration:<id>:live`. Preserve dry-run evidence in separate dry-run fields or metadata, and require `promote_dry_run=true` plus matching planned target/intent digest before a planned dry-run row can transition to live `running`.
- SYMPHONY ANCHOR: Whole-System Traversal: “Every step should have a local source owner, a local regression or dry-run proof where possible...” API, Webhook, And Extension Contracts: “Action-worker contracts that define actor, action, target, reason, status, audit, dry-run, confirmation, retry, timeout, and rollback fields.”
- RATIONALE: Operators get one stable migration identity tying plan, dry-run proof, live execution, audit, and rollback together, while idempotency remains phase-safe. Accidental live reuse of a dry-run ID without promotion fails closed. The Captain still owns the Pod outcome, not host mutation policy.
- TRADEOFFS & ALTERNATIVES: Reject removing `dry_run` from the intent digest because that can replay a dry-run as if it were live-safe. Reject forcing a new migration ID for live because it splits evidence and makes surfaces reconcile two truths. Residual risk is old dry-run planned rows; handle them with a clear “rerun dry-run or use new migration ID” upgrade error.
- EFFORT / BLAST-RADIUS: medium; touches migration idempotency semantics, action-worker reprovision metadata in [python/arclink_action_worker.py](/root/arclink/python/arclink_action_worker.py:1223), migration schema/metadata, audit/action links, and pod-migration/action-worker tests.

### DECISION 3: Rollback lifecycle terminal status
- RECOMMENDATION: Define `rolled_back` as proven rollback only. `_rollback_lifecycle` currently catches teardown/restart failures into metadata in [python/arclink_pod_migration.py](/root/arclink/python/arclink_pod_migration.py:707), and `_mark_rollback` always writes `status = 'rolled_back'` in [python/arclink_pod_migration.py](/root/arclink/python/arclink_pod_migration.py:861). Change it so target teardown failure or required source restart failure sets migration `status='failed'`, emits `pod_migration_rollback_failed`, records redacted lifecycle metadata, marks pod-migration health failed, and preserves the capture directory for repair. Use `rolled_back` only when required rollback steps are successful or no-op.
- SYMPHONY ANCHOR: Fleet, Provisioning, Ingress, And Recovery: “Rollback preserves state by default and only deletes volumes with explicit destructive metadata and confirmation.” Notifications, Incidents, And Evidence: “ArcLink should never fail silently.”
- RATIONALE: A failed source restart or target teardown is not a successful rollback; calling it `rolled_back` creates a false system truth. Using existing `failed` avoids adding a new public enum while still failing closed and leaving operator-visible evidence and repair material.
- TRADEOFFS & ALTERNATIVES: Reject keeping best-effort `rolled_back` because it hides host/process uncertainty. A new `rollback_failed` enum would be clearer but has larger API/web/bot compatibility cost; `failed` plus structured `rollback_metadata_json.rollback_outcome` is the safer first contract. Residual risk is that callers must inspect metadata for “source restored but target cleanup failed.”
- EFFORT / BLAST-RADIUS: medium; touches `arclink_pod_migration`, action-worker expectations, dashboard/API/Raven status rendering, service-health evidence, and rollback tests.
