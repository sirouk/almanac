## CANON-19 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Backup deploy-key private-ref rail has no consumer
- RECOMMENDATION: Preserve the public dashboard contract, but make the rail real. `python/arclink_dashboard.py:1111-1113` creates `server_state:agent-backup-deploy-key:<digest>`, `:1125-1147` writes the private key, and `:1266-1275` stores the ref; today verification still fails closed (`:1396-1419`) and the worker only records `failed_closed` (`python/arclink_action_worker.py:1026-1059`), while shipped backup uses `AGENT_BACKUP_KEY_PATH` (`bin/backup-agent-home.sh:39,199-202`). Add a CANON-22 resolver/verifier that maps only that ref prefix to the configured private staging root, rejects missing/symlink/permission-bad keys, runs GitHub read plus dry-run write only under `PG-BACKUP`, and activates backup only after it writes the actual backup state consumed by backup scripts. Existing orphaned metadata should migrate to inactive/failed-closed evidence, not be deleted silently.
- SYMPHONY ANCHOR: Backup, Restore, And Data Lifecycle — “private backup repo activation only after read and dry-run write checks pass”; Secrets, Keys, And Rotation — “Secret references move through provisioning instead of plaintext values.”
- RATIONALE: This keeps state and client fields stable while restoring a single source owner for private key material. Captains can request setup, but Operators own backup secrets and live proof. The current fail-closed behavior remains when `PG-BACKUP` is not armed; successful activation becomes evidence-backed instead of metadata-only.
- TRADEOFFS & ALTERNATIVES: Removing the field is cleaner but breaks the public backup setup contract. Leaving it orphaned preserves compatibility but keeps unused private keys on disk. Reusing the separate per-agent `$HOME/.ssh/arclink-agent-backup-ed25519` lane without a migration would create two truths.
- EFFORT / BLAST-RADIUS: high; touches dashboard metadata, action worker, CANON-22 backup scripts, migration/audit rows, web/Raven backup status copy, and `tests/test_arclink_dashboard.py` plus `tests/test_arclink_action_worker.py`.

### DECISION 2: Docker/domain dashboard SSO is silently generated and enabled
- RECOMMENDATION: Make dashboard SSO explicit operator policy, default disabled for new/reconfigured deployments. Current Docker code generates a per-user secret (`bin/arclink-docker.sh:1696-1718`), mounts it (`:1915-1922`), passes it to the installer (`:2044-2047`), and the installer writes `sso_session_secret`/`sso_subject` (`bin/install-deployment-hermes-home.sh:92-103,163-171`); the proxy then accepts the SSO cookie (`python/arclink_dashboard_auth_proxy.py:672-700`) and scopes it to `/` plus optional domain (`:729-735`). Add `ARCLINK_DASHBOARD_SSO_MODE=disabled|captain_account`, require `sso_enabled:true` in the access file before issuing or accepting SSO cookies, preserve existing secret files unused unless the Operator enables/rotates them, and surface `disabled/enabled/legacy_secret_present/revoked` as redacted status.
- SYMPHONY ANCHOR: Identity, Access, And Session Governance — “Session cookies... each have their own lifetime, storage location, revocation path, and audit behavior”; North Star — “Operators own the universe: hosts, secrets, fleet, policy...”
- RATIONALE: Cross-dashboard SSO is reasonable for one Captain account, but not as a silent default. Opt-in keeps the convenience path while making the expanded cookie boundary an operator-owned policy with revocation, status, local tests, and `PG-HERMES-DASHBOARD-SSO` live proof.
- TRADEOFFS & ALTERNATIVES: Always-on SSO is smoother but weakens fail-closed policy. Removing SSO entirely loses an intended same-account Crew dashboard flow. Disabling existing SSO may require users to log in per dashboard once, but scoped dashboard login remains available and state is not destroyed.
- EFFORT / BLAST-RADIUS: med; touches Docker provisioning, `install-deployment-hermes-home.sh`, auth proxy, generated config/docs, dashboard/Raven status, and SSO proxy/Docker regression tests.

### DECISION 3: Auth-proxy login throttle is process-local
- RECOMMENDATION: Replace the production throttle store with durable SQLite, defaulting to an `arclink-dashboard-login-rate-limits.sqlite3` beside `arclink-web-access.json`, with an override for operator dashboard deployments. Current throttle is `_LOGIN_FAILURES` in memory (`python/arclink_dashboard_auth_proxy.py:63-66`) with IP/user/combo keys (`:368-405`) and is enforced on login (`:1059-1080`). Reuse the hosted API pattern of `rate_limits(scope, subject, observed_at)` (`python/arclink_control.py:728-733`; `python/arclink_api_auth.py:460-523`): `BEGIN IMMEDIATE`, prune by window, scope by deployment/prefix/realm plus IP/user/combo, and fail closed with no session cookie if the required store cannot be opened.
- SYMPHONY ANCHOR: Identity, Access, And Session Governance — “Rate limits, replay protection... should be consistent enough that the same action cannot be made safer or more dangerous merely by choosing chat instead of dashboard.”
- RATIONALE: A local durable store survives restarts and multiple proxy processes without giving every Pod proxy write access to the central control DB. It keeps the login boundary local, auditable, and fail-closed, while matching the system’s existing rate-limit schema.
- TRADEOFFS & ALTERNATIVES: A central control-DB throttle gives one global truth but increases the privilege of every dashboard proxy and complicates remote Pods. Keeping memory as production default is simpler but loses protection on restart. A memory mode can remain only as explicit test/dev mode.
- EFFORT / BLAST-RADIUS: med; touches auth proxy, proxy launcher defaults, access-file-adjacent state permissions, and proxy throttle tests including restart/multiprocess/unavailable-store cases.

### DECISION 4: Env-derived dashboard hosts need an allowed-host policy
- RECOMMENDATION: Add a single URL/host policy owner, preferably in `python/arclink_adapters.py`, and make dashboard URL readers consume it. Current `_deployment_urls` builds URLs from deployment metadata and env (`python/arclink_dashboard.py:825-911`), stored URLs only require `https://` (`:883-891`), and `_control_notion_webhook_public_url` accepts explicit/env hosts (`:949-974`). Canonical ingress already narrows the intended shape: domain mode uses `u-{prefix}.{base_domain}` and `hermes-{prefix}.{base_domain}` (`docs/arclink/ingress-plan.md:25-36`), Tailscale path mode uses `https://{tailscale_dns_name}/u/{prefix}` (`docs/arclink/ingress-plan.md:46-56,138-140`). Encode that: domain hosts must match generated ArcLink hostnames under configured base/allowed suffixes; Tailscale hosts must match configured Tailscale/control hosts; ports are allowed only from persisted tailnet service ports; no userinfo/query/fragment/path escape. Invalid env or metadata should omit/block URLs and report `blocked_invalid_host_policy`, while install/reconfigure should fail before activation.
- SYMPHONY ANCHOR: Fleet, Provisioning, Ingress, And Recovery — “Ingress is either domain... or Tailscale path routing, with clear teardown evidence”; Configuration, Schema, And Migration — generated config detects “stale, missing, deprecated, or incompatible values before services start.”
- RATIONALE: Operators own ingress policy, but operator env is still input and should not create unsafe Captain-facing links. A central validator keeps dashboard, provisioning, Notion callback, web hrefs, and live ingress proof aligned with the same system truth.
- TRADEOFFS & ALTERNATIVES: Validating only in the web client is too late because Raven/API would still emit bad links. Rejecting all explicit URLs would break legitimate Tailscale/control callback overrides. The residual risk is migration noise from old metadata, so make read paths block unsafe links without deleting metadata and require reconfigure to repair.
- EFFORT / BLAST-RADIUS: med; touches adapters/ingress policy, dashboard read model, provisioning/Docker URL materialization, hosted web link assumptions, ingress docs, and regression tests for valid domain, valid Tailscale, stored unsafe URL, and bad env fail-closed cases.
