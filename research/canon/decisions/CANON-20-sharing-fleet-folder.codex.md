## CANON-20 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Full hub URL host/scheme allowlist for remote fleet-share hubs
- RECOMMENDATION: Keep operator-provided remote SSH hubs, but make them explicit policy. Add one shared validator used by `python/arclink_fleet_share.py` and `python/arclink_provisioning.py`: local refs must stay under `ARCLINK_FLEET_SHARE_HUB_ROOT`; remote refs must be `ssh://...` or scp-style `user@host:path`, match an operator allowlist derived from private config, and use a known_hosts-backed `GIT_SSH_COMMAND` with fail-closed host checking. Continue rejecting git remote-helper syntax. Disallow `https`, `file`, `git`, and unknown schemes unless a later credential design explicitly adds them. Add local tests around `_assert_safe_git_arg` / `_clean_git_ref` gaps and a named live gate `PG-FLEET-SHARE-HUB` that proves `ls-remote` plus a redacted write/read/delete sentinel against the configured hub.
- SYMPHONY ANCHOR: North Star: "Operators own the universe: hosts, secrets, fleet, policy"
- RATIONALE: Current code only blocks option/control/remote-helper syntax in `_assert_safe_git_arg` and `_clean_git_ref`, then accepts remote refs via `ensure_hub_repo(... ls-remote ...)` (`python/arclink_fleet_share.py:125-140,225-242`; `python/arclink_provisioning.py:105-116,581-609`). The operator should own which hub host is valid, but ArcPods should fail closed if rendered or runtime hub refs drift to an unapproved host.
- TRADEOFFS & ALTERNATIVES: Hard-coding only the control WireGuard host was rejected because production explicitly supports operator-run remote SSH hubs. Allowing arbitrary SSH URLs was rejected because a pod/env/config error can redirect Captain Fleet contents. HTTPS was rejected for now because it needs a separate token storage/redaction/rotation story.
- EFFORT / BLAST-RADIUS: med; touches fleet-share validation, ArcPod compose rendering, fleet-share SSH env, tests, and live-proof tooling.

### DECISION 2: Distributed fleet-share sync locking and bounded retry policy
- RECOMMENDATION: Do not introduce a cross-machine distributed lock into the current git sync path. Keep git as the convergence authority, but replace the fixed `range(2)` loop with a named bounded policy: e.g. `ARCLINK_FLEET_SHARE_SYNC_MAX_ATTEMPTS=4`, short capped backoff/jitter, no force-push, and terminal `error` only after preserving local commits. Add a local per-working-copy nonblocking lock to prevent same-pod overlap. Surface retry exhaustion through `last_sync_status/detail` and local status JSON so Drive/Code/Raven/dashboard can show one truth. Add `PG-FLEET-SHARE-SYNC` for live two-ArcPod convergence/conflict proof.
- SYMPHONY ANCHOR: Sharing: "multi-writer convergence and conflict surfacing"
- RATIONALE: Current `sync_member` commits, fetches, rebases, pushes, and after two attempts returns `error` (`python/arclink_fleet_share.py:341-409`). That is safe against clobbering, but under contention it can report failure on a healthy hub. A bounded retry policy improves reliability without adding a broker whose failure mode could block all sharing.
- TRADEOFFS & ALTERNATIVES: A true distributed lock was rejected for this piece because it needs a hub-side lease/broker, stale-lock recovery, and remote proof. If ArcLink later needs strict serialization, build it as an explicit control-owned sync broker with lease expiry and evidence, not as ad hoc local locking.
- EFFORT / BLAST-RADIUS: med; touches `sync_member`, fleet-share status recording, tests, and cross-surface status display. No schema change required if `error/conflict/synced` remain the public result states.

### DECISION 3: Dead `paused` share and member `pending` statuses
- RECOMMENDATION: Remove them from the active CANON-20 contract rather than activating them now. Change fleet-share table checks/constants to `active|removed` for shares and members, after an upgrade preflight confirms no legacy `paused`/`pending` rows. If such rows exist, fail closed with a clear operator repair command instead of silently mapping them. Do not add a pause lifecycle until there is an enforceable sync broker, key revocation path, or pod-service control path that actually stops in-pod `fleet-share-sync`.
- SYMPHONY ANCHOR: Whole-System Traversal: "state by default and leave redacted evidence"
- RATIONALE: Current schema allows `paused` and member `pending` (`python/arclink_control.py:1168-1187,3255-3256`), but fleet-share code writes only `active`/`removed` and in-pod sync does not consult the DB before pushing (`python/arclink_fleet_share.py:489-657,870-909`). Keeping unenforced statuses creates a false system truth; activating them without enforcement would be worse.
- TRADEOFFS & ALTERNATIVES: Leaving the dead statuses was rejected because surfaces could claim a state the sync engine ignores. Activating `paused`/`pending` was rejected for now because it requires new API/action-worker/plugin semantics and hard enforcement across existing pods. Removing them is a contract change, but it is the smallest path to same-truth behavior.
- EFFORT / BLAST-RADIUS: med; touches schema/migration preflight, status constants, diagnostics, fleet-share tests, and any docs/OpenAPI/status fixtures that expose fleet-share member states.
