### DECISION 1: Whether org-profile fan-out should include non-`role='user'` or inactive agents
- RECOMMENDATION: Keep fan-out limited to active user agents. Preserve `_active_agent_rows()` as `WHERE a.role = 'user' AND a.status = 'active'` (`python/arclink_org_profile.py:2191-2205`) and the matching refresh gate in `signal_agent_refresh_from_curator()` (`python/arclink_control.py:19171-19185`). Codify the contract with a regression that non-user and inactive rows do not get slices, stale-slice deletion, SOUL writes, or refresh triggers. Optionally rename the helper/report field to `active_user_agent_fanout` so the boundary is visible in `apply`/`doctor`.
- SYMPHONY ANCHOR: Pods, Isolation, And SOUL: “An ArcPod is the Captain's private deployment”; “Crew Training should … project an additive SOUL overlay and role/context slices without rewriting historical memory.”
- RATIONALE: The current scope matches the ownership model: Captains own Pods and Crew, while Operators own host, fleet, secrets, policy, and upgrade surfaces. `role='user'` is the enrolled Crew/Pod lane; non-user agents such as Curator/operator services should get any operator context through an explicit operator-facing surface, not automatic Crew SOUL fan-out. Inactive agents should preserve state by default and not receive fresh policy/context writes. This fails closed: excluded agents get no new org-profile material rather than accidental authority, privacy, or host-policy leakage.
- TRADEOFFS & ALTERNATIVES: Widening to all active agents was rejected because it blurs Captain Crew context with operator/admin runtime context. Widening to inactive agents was rejected because it mutates preserved or retired state. A future operator-org-profile summary is acceptable, but it should be a separate RBAC/audit contract backed by `applied.json`, not a change to Crew fan-out.
- EFFORT / BLAST-RADIUS: Low. Touches `python/arclink_org_profile.py`, `docs/org-profile.md`, and focused org-profile tests; medium only if a new operator-facing context surface is added.
