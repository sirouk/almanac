## CANON-22 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Quiet-hours local/DST semantics need a timezone/config contract
- RECOMMENDATION: Use the existing operator-owned `ARCLINK_ORG_TIMEZONE` as the canonical timezone for `ARCLINK_ORG_QUIET_HOURS`; do not add a Wrapped-only knob. `bin/deploy.sh` already defaults and validates it (`Etc/UTC`, IANA validation, persisted config), while `python/arclink_wrapped.py` currently ignores it and applies quiet-hour wall times to UTC (`_parse_dt` normalizes to UTC; `next_attempt_after_quiet_hours` uses `current.replace(...)`). Implement `ZoneInfo` in `next_attempt_after_quiet_hours`, convert UTC `now` to local time, compute the quiet boundary locally, then store `notification_outbox.next_attempt_at` back in UTC. If quiet hours are configured but malformed, or the timezone is invalid, fail closed by raising `ArcLinkWrappedError` instead of delivering immediately. Add local tests for UTC default, `America/New_York` summer/winter, an overnight DST-crossing window, and invalid timezone/malformed quiet-hours refusal.
- SYMPHONY ANCHOR: `Configuration, Schema, And Migration` — “Generated config includes enough version/release context to detect stale, missing, deprecated, or incompatible values before services start.”
- RATIONALE: This makes the operator’s runtime config the single source of truth: `ARCLINK_ORG_TIMEZONE` is collected and persisted by deploy, `ARCLINK_ORG_QUIET_HOURS` is explicitly described as local time, and Wrapped delivery should obey that same contract. It preserves UTC storage while making policy evaluation local/DST-correct. It also fails closed: bad local-time policy blocks queueing and leaves scheduler failure evidence rather than sending during a quiet window.
- TRADEOFFS & ALTERNATIVES: Rejected keeping UTC-only semantics because it contradicts the local-time prompt and Captain-facing SOUL context. Rejected per-Captain timezone for this piece because current Wrapped delivery is org policy, not per-user scheduling. Residual risk: actual Telegram/Discord send behavior remains downstream `PG-BOTS`; this decision only fixes local scheduling.
- EFFORT / BLAST-RADIUS: med. Touches `python/arclink_wrapped.py`, Wrapped tests, and possibly deploy/config docs; no DB migration required.

### DECISION 2: Backup reconcile needs script-owned single-writer locks
- RECOMMENDATION: Make the mutating backup scripts the lock owners. Add non-blocking `flock` around the full local-repo mutation and reconcile/push critical section in both lanes. Use explicit private state paths: shared lane `ARCLINK_BACKUP_LOCK_FILE=${ARCLINK_BACKUP_LOCK_FILE:-$STATE_DIR/locks/arclink-priv-backup.lock}`; agent lane `AGENT_BACKUP_LOCK_FILE=${AGENT_BACKUP_LOCK_FILE:-$HERMES_HOME_TARGET/state/agent-home-backup/.backup.lock}`. Move the existing agent-home cron lock out of the generated wrapper in `bin/install-agent-cron-jobs.sh` or make the wrapper delegate to the script, so there is one authoritative lock, not nested competing locks. If lock is held, make no git/local/remote mutation, write a redacted `busy` status, and return a scheduled-no-op result. Explicitly exclude lock files from any staged backup content when under `ARCLINK_PRIV_DIR`.
- SYMPHONY ANCHOR: `Backup, Restore, And Data Lifecycle` — “Backup cannot mean ‘files were copied once.’” Also `North Star` — “Upgrades, backups, restore... preserve state by default and leave redacted evidence.”
- RATIONALE: Current shared backup relies on the systemd timer comment as the single-writer guarantee, while `backup-to-github.sh` can still be run manually and can archive/force-align unrelated history. Agent backups are locked by the Hermes cron wrapper, but direct `backup-agent-home.sh` runs bypass that ownership. The script is the local source owner for the local repo and remote branch mutation, so the script should serialize all entrypoints. This preserves state, prevents double archive/force races, and gives health/Raven/status surfaces one truth.
- TRADEOFFS & ALTERNATIVES: Rejected relying on systemd/Hermes scheduling because repair/manual paths bypass it. Rejected blocking locks because a hung backup can stall cron/timers indefinitely. Rejected a distributed remote lock for now; this is a host-local guarantee, and cross-host use of the same backup remote should remain operator-disallowed unless a future brokered lease is designed. Live recoverability remains `PG-BACKUP`.
- EFFORT / BLAST-RADIUS: med. Touches `bin/common.sh` or local shell helpers, `bin/backup-to-github.sh`, `bin/backup-agent-home.sh`, `bin/install-agent-cron-jobs.sh`, backup regression tests, and possibly health/status copy.
