## CANON-25 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: `agent-process-helper` needs outbound egress without exposing its root helper listener on that egress path
- RECOMMENDATION: Keep `agent-process-helper-egress-net` non-internal and single-service for outbound runtime work, but move the helper API off TCP networking. Replace `ARCLINK_AGENT_PROCESS_HELPER_URL=http://agent-process-helper:8916` with a Unix socket on a narrow shared runtime volume mounted only into `agent-process-helper` and `agent-supervisor`; keep token auth as defense-in-depth; bind any remaining TCP health listener to `127.0.0.1` only or healthcheck the Unix socket directly. Remove the `0.0.0.0` Compose listener at `compose.yaml:931` and update `arclink_docker_agent_supervisor.py`, `arclink_agent_process_helper.py`, `compose.yaml`, authority inventory, and tests so missing/unsafe socket means no process action.
- SYMPHONY ANCHOR: `Pods, Isolation, And SOUL` — “Terminal and process execution stay behind bounded broker/helper contracts” and host root is “a high-trust boundary until stronger isolation is implemented.”
- RATIONALE: The current code binds the root helper to `0.0.0.0` while also attaching it to non-internal egress. A Unix-socket request lane lets spawned agent processes retain egress while the helper command surface is reachable only by the source-owned supervisor. This preserves runtime behavior, narrows authority, and fails closed without relying on Docker bridge accidentals.
- TRADEOFFS & ALTERNATIVES: Simply marking the egress network `internal: true` breaks outbound agent work. Binding to a static internal-net IP is brittle and Docker-IPAM dependent. Source-IP filtering is a useful interim hardening but still leaves a listener exposed. Residual risk remains the trusted root process boundary itself; prove with local socket/permission regressions and `PG-HERMES` live proof for real agent process work.
- EFFORT / BLAST-RADIUS: high; touches Compose topology, helper transport, supervisor client, healthchecks, authority inventory, docs, and Docker regression tests.

### DECISION 2: `docker-job-loop.sh` should not globally crash containers on every child failure
- RECOMMENDATION: Do not make child failure globally exit the loop. Codify the current default as `failure_policy=continue`: `run_job_once` captures the child rc, writes redacted `status=fail`/`exit_code`, and `docker-health.sh` fails the Control Node health view. Add an explicit per-job option such as `--failure-policy continue|exit|exit-after=N`, with any `exit` use declared in Compose per job. Update tests so default failures remain owner-visible state, while opt-in exit policies exit only after writing final evidence.
- SYMPHONY ANCHOR: `Notifications, Incidents, And Evidence` — “Every important background path should have an owner-visible state, a retry or repair path, and evidence that can be shared without secrets.”
- RATIONALE: These are recurring pollers and repair loops, not one-shot daemons. A blanket crash would change retry cadence across jobs such as notification delivery, inventory, qmd, backup, and health-watch, and could create restart storms without repairing the underlying cause. The source-owned contract should be state and evidence first, process restart only where explicitly useful.
- TRADEOFFS & ALTERNATIVES: A global `exit $rc` is simpler and uses Docker restart semantics, but it hides job-specific policy inside supervisor behavior. Continuing forever risks a container showing “running” while work is failing, so health/status consumers must remain strict and complete. The residual risk is missed status coverage for optional/profile jobs; close that with tests that every active `docker-job-loop` service has a health/evidence consumer.
- EFFORT / BLAST-RADIUS: med; touches `bin/docker-job-loop.sh`, Compose job declarations if policies are used, `bin/docker-health.sh` coverage, and tests.

### DECISION 3: `control health` should stop silently repairing Nextcloud data permissions
- RECOMMENDATION: Split read-only health from mutating repair. Replace the `health()` call to `repair_running_nextcloud_data_dir` with a read-only `check_running_nextcloud_data_dir` that fails with the exact repair command when `.ncdata`, `index.html`, owner, or mode are wrong. Add an explicit operator command such as `./deploy.sh control repair nextcloud-data` that runs the existing repair, supports dry-run where practical, and writes redacted evidence under private state. Install/upgrade may call the repair as a clearly mutating pre-health step.
- SYMPHONY ANCHOR: `Backup, Restore, And Data Lifecycle` — recoverability must be proven “without surprising the Captain or destroying state.”
- RATIONALE: Current `health()` mutates `/var/www/html/data` and suppresses repair errors before probing Nextcloud. That makes a diagnostic command alter runtime state and can hide drift. An explicit repair keeps recovery behavior available, makes the operator the action owner, preserves state by default, and leaves evidence.
- TRADEOFFS & ALTERNATIVES: Removing the repair outright could regress live recovery for harmless permission drift. Keeping it hidden keeps health convenient but violates same-truth diagnostics. The staged split preserves repair while making health fail closed and honest.
- EFFORT / BLAST-RADIUS: med; touches `bin/arclink-docker.sh`, `deploy.sh` command routing, docs/runbook, and health/Nextcloud regression tests.

### DECISION 4: Embedded and standalone Nextcloud should not share an implicit fresh-install port default
- RECOMMENDATION: Keep embedded Control Node Nextcloud on `NEXTCLOUD_PORT:-18080`, but give the standalone legacy sidecar its own explicit port key, e.g. `ARCLINK_STANDALONE_NEXTCLOUD_PORT`, defaulting to `18081` for new configs. Preserve old operators by honoring an already-explicit `NEXTCLOUD_PORT` or `ARCLINK_STANDALONE_NEXTCLOUD_PORT`. Update `compose/nextcloud-compose.yml`, `bin/nextcloud-up.sh`, `rotate-nextcloud-secrets.sh`, common env export, docs, and tests. Add preflight collision checks so either stack fails closed with operator copy if the chosen port is already owned by the other stack.
- SYMPHONY ANCHOR: `Configuration, Schema, And Migration` — “Reconfigure is safe for changing ports… without silently deleting runtime state.”
- RATIONALE: The Control Node is the canonical public lane, so its `18080` contract should remain stable. The standalone file is legacy/migration tooling; giving it a separate source-owned default avoids a same-host collision while preserving explicit existing state. The same-truth surface becomes “embedded port” versus “standalone port,” not two stacks silently racing for one bind.
- TRADEOFFS & ALTERNATIVES: Leaving both defaults at `18080` preserves legacy habit but fails late at bind time. Changing global `NEXTCLOUD_PORT` would break the canonical embedded lane. Some old configs may still explicitly request `18080`; the collision preflight must catch that and tell the operator exactly which port to change.
- EFFORT / BLAST-RADIUS: med; touches standalone Nextcloud scripts, Compose, common config defaults, deploy prompts/docs, and port regression tests.
