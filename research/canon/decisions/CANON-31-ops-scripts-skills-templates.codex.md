## CANON-31 — Codex (GPT-5.5 xhigh) decision recommendations (symphony-anchored)

### DECISION 1: Retire `ENABLE_TAILSCALE_SERVE` as an active shared Nextcloud/MCP contract
- RECOMMENDATION: Rename/remove it from the active contract via a compatibility migration, not a blind delete. Make `ENABLE_TAILSCALE_SERVE` a deprecated, ignored legacy key for shared Nextcloud/internal MCP Serve. Introduce an explicit narrow flag such as `ARCLINK_AGENT_DASHBOARD_TAILSCALE_SERVE_ENABLED` for the still-real per-agent dashboard Serve path now driven by `ARCLINK_AGENT_ENABLE_TAILSCALE_SERVE` (`python/arclink_agent_access.py:552-555`, `python/arclink_enrollment_provisioner.py:1137-1139`). Accept the old agent flag as a one-release alias only when the new flag is absent; do not default it from `ENABLE_TAILSCALE_SERVE` (`python/arclink_control.py:525-528`, `config/arclink.env.example:143-144`). Remove the legacy install prompts for “Tailscale HTTPS proxy for Nextcloud” and “internal MCP routes” (`bin/deploy.sh:4787`, `4809`), stop writing `ARCLINK_AGENT_ENABLE_TAILSCALE_SERVE="$ENABLE_TAILSCALE_SERVE"` (`bin/deploy.sh:2445`), and stop synthesizing qmd/control MCP tailnet URLs from DNS or the retired flag (`bin/deploy.sh:1578-1586`, `1605-1613`). Keep `ARCLINK_INGRESS_MODE=tailscale` plus `ARCLINK_TAILSCALE_*` as the canonical Control Node ingress contract. Turn `bin/tailscale-nextcloud-serve.sh` into a nonzero retired-command shim; leave `tailscale-nextcloud-unserve.sh` as the explicit cleanup command. Add config-migration evidence and tests proving old configs preserve explicit agent-dashboard publication but no longer imply shared Nextcloud/MCP exposure.
- SYMPHONY ANCHOR: **Third-Party Integration Boundaries** — “Cloudflare and Tailscale own ingress primitives; ArcLink owns desired-state records, teardown evidence, proof gates, and clear domain/Tailscale mode selection.” Also **Configuration, Schema, And Migration** — “Generated config includes enough version/release context to detect stale, missing, deprecated, or incompatible values before services start.”
- RATIONALE: The current code already says shared Nextcloud/internal MCP Serve is retired (`bin/tailscale-nextcloud-serve.sh:235-240`, `bin/deploy.sh:397-400`), but the config, prompts, health, and resource rendering still let the old flag look active. That violates same-truth and can hand agents false “expected” tailnet rails. The migration preserves real state by carrying forward only the explicit agent-dashboard publication intent, while failing closed for shared MCP/Nextcloud exposure unless a new explicit owner and proof gate exists. External Tailscale readiness should remain under `PG-INGRESS`, not inferred from MagicDNS or a stale env var.
- TRADEOFFS & ALTERNATIVES: Rejected hard deletion in one release because it could break existing per-agent dashboard URLs. Rejected keeping `ENABLE_TAILSCALE_SERVE` as an alias because it keeps a retired host-level exposure policy alive under a misleading name. Rejected restoring shared Nextcloud/MCP Serve because Drive/Code now belong behind the authenticated Hermes dashboard and Captains must not own host/internal rails.
- EFFORT / BLAST-RADIUS: high — touches deploy prompts/config writing, `Config.from_env`, agent access publishing, onboarding/resource-map text, health checks, env examples, legacy Tailscale scripts, and regression tests (`test_deploy_regressions`, `test_arclink_agent_access`, onboarding/resource/plugin tests).
