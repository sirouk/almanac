# CANON-06 — Curator Onboarding

## PIECE
This piece is the **Curator (the "Console Curator" / Captain-guide persona, runtime-named "Raven, Curator of the Console")** private-DM onboarding surface and its operator-control overlay, for **Telegram** and **Discord**, plus the shell/systemd entrypoints that run the Curator's own Hermes gateway, recovery TUI, refresh job, and one-time bootstrap. It owns exactly eight tracked files:
- `python/arclink_curator_onboarding.py` (1263 lines) — Telegram long-poll worker. Polls `getUpdates`, routes operator vs Captain messages, handles inline-button callbacks, registers command menus, and offset/failure bookkeeping.
- `python/arclink_curator_discord_onboarding.py` (1172 lines) — Discord `discord.py` gateway client. Registers slash commands, handles DM onboarding, operator-channel control, and component-button interactions.
- `bin/curator-gateway.sh` (162 lines) — launches the Curator's user-facing Hermes `gateway`, with env/token de-conflict against the onboarding workers.
- `bin/curator-tui.sh` (33 lines) — "godmode" recovery: `exec hermes` against the Curator Hermes home (always available, cannot be disabled).
- `bin/curator-refresh.sh` (10 lines) — wrapper for `arclink_ctl.py internal curator-refresh` (hourly timer).
- `bin/bootstrap-curator.sh` (1079 lines) — interactive/headless one-time setup: model preset, channel choice, gateway setup, operator-notify channel, registration, systemd enable/disable.
- `bin/arclink-curator-onboarding.sh` / `bin/arclink-curator-discord-onboarding.sh` (10 lines each) — thin `exec python3 …` launchers used by the systemd units.

Both Python workers are **thin transport adapters**: every real onboarding state transition, copy generation, secret scrubbing, request queueing, and DB write of session state lives in the adjacent CANON-04 modules (`arclink_onboarding_flow.py`, `arclink_onboarding_completion.py`) which they import (`arclink_curator_onboarding.py:35-48`, `arclink_curator_discord_onboarding.py:30-44`). This piece's own logic is: transport plumbing, operator allow/deny gating, callback ownership checks, command-menu registration, the Telegram failure-skip ledger, and the systemd/shell lifecycle.

Found-but-not-listed adjacent files I did **not** claim (they belong to other pieces): `systemd/user/arclink-curator-*.service|.timer` (CANON-26), `tests/test_arclink_curator_onboarding_regressions.py` + `tests/test_bootstrap_curator_regressions.py` (CANON-29), `docs/curator-onboarding-transcript-notes.md` + `curator_discord_avatar.png` (CANON-32). I read the systemd units and tests to verify contracts but list them only as touch points.

## CODE-PATH TRACE (Telegram operator one-tap upgrade-install, end to end)
1. systemd `arclink-curator-onboarding.service` → `bin/arclink-curator-onboarding.sh:10` `exec python3 arclink_curator_onboarding.py`.
2. `main()` gate `curator_telegram_onboarding_enabled` true; token resolved; `getMe` → `curator_bot_id`; `register_telegram_bot_commands` writes menus (`:1222-1234`).
3. Loop `run_once` reads offset `get_setting(conn, OFFSET_SETTING_KEY)` (`:1167`), calls `telegram_get_updates(offset, timeout)` (`:1169`).
4. For each update → `process_update` (`:1175`). `callback_query` present → `_handle_user_completion_callback`/`_backup`/`_notion` return False (data not theirs) → `_handle_operator_callback` (`:1131`).
5. `_handle_operator_callback` checks `data.startswith("arclink:")` (`:807`) and `_operator_sender_allowed` (`:809`) → on fail answers "restricted" alert (`:815-821`).
6. `data.split(":",3)` → `scope=upgrade, action=install, target_id=<sha>` (`:857`). `one_tap_install` true iff `action=="install"`, scope in {upgrade,pin-upgrade}, and `operator_button_approvals_enabled` (`:873`).
7. With approval code configured + `one_tap_install` → re-issues `/upgrade` menu via `dispatch_operator_raven_command` (`:877`) and returns (two-tap nonce path). Without code → directly `dispatch_operator_raven_command(conn, "/upgrade confirm", idempotency_key=callback_query_id)` (`:981`).
8. Raven result `{message, buttons}` (CANON-14) → `replacement_text` built (`:989`) → `_replace_operator_callback_message` edits the Telegram message (`:1040`) → `telegram_answer_callback_query` toast (`:1043`).

Failure variant: any exception in `process_update` → `record_onboarding_update_failure` (`:1183`); if `failure_count >= onboarding_update_failure_limit` → `mark_onboarding_update_skipped` + advance offset + operator notice (`:1185-1208`); else `notify_operator_worker_failure` + `break` (do not advance — poison-pill retry until limit) (`:1190-1198`).

## CODE vs COMMENT/DOC/NAME DRIFT
1. **`arclink_upgrade_last_dismissed_sha` is written but never read.** Written at `curator_onboarding.py:975` and `curator_discord_onboarding.py:272`; a full-tree search finds **zero** readers (no `get_setting`/SELECT on that key). The actual upgrade-notice suppression key is `arclink_upgrade_last_notified_sha` (read in `arclink_ctl.py:1822`, per DISSECT.md:277). So tapping "Dismiss" persists a value that changes nothing — the result text "Dismissed ArcLink update notice…" overstates the effect. This matches DISSECT.md's INFO finding (`:295`).
2. **Prior doc `04-captain-bots-onboarding.md` mislabels this whole subsystem as the "OLD curator/Almanac" path and implies it is undocumented/legacy** (`:17`, `:134`). Code shows it is **actively wired** as the live curator surface (systemd units, bootstrap enable/disable, GAP-029 operator overlay, one-tap upgrade buttons newer than that doc). The doc also lists operator Telegram commands as 12 (`operator_status…academy_status`, `:122`); the actual registered `TELEGRAM_OPERATOR_COMMANDS` set here is **23 commands** (`curator_onboarding.py:83-107`), and Discord registers a comparably large operator slash set (`:699-901`). The prior doc undercounts.
3. **Persona-name drift**: argparse describes the worker as "for Raven, Curator of the Console" (`:116`) while slash-command descriptions and refusal copy say "Curator" (`discord:558,935`). Both names refer to the same bot; the runtime display name is resolved elsewhere. Cosmetic, but the doc's "Curator vs Raven" confusion (`04-…:158`) is real in the code copy too.
4. **`has_curator_non_telegram_gateway_channels` is a misnomer** — it is a 1-line alias that calls `has_curator_non_onboarding_gateway_channels` (`common.sh:761-763`), i.e. it covers Discord-onboarding too, not just "non-telegram". Name lies; behavior is "non-onboarding".
5. **Discord `/upgrade` command help says "host upgrade/repair"** (`:625`) but the bare (non-confirm) path renders the Raven `/upgrade` *menu* (`:632-644`); only `confirm` dispatches. Minor: the description undersells the preview/menu behavior.

## ADVERSARIAL SELF-CHECK
1. **Claim: the dismissed-sha key is truly never read.** Falsifier: a reader using string concatenation or a different module (e.g. dashboard) that I did not grep. I searched `--type py` for `last_dismissed_sha`/`last_dismissed` excluding the writers and found nothing, and DISSECT.md independently agrees — but a dynamically-built key name would evade this.
2. **Claim: `_claim_discord_message_once` grows `settings` unbounded.** Falsifier: a sweeper/TTL job elsewhere deleting `curator_discord_onboarding_seen_message:*` rows. I did not find one, but I did not exhaustively grep every janitor. If absent, this is a slow table-bloat leak.
3. **Claim: the two workers and the gateway never double-bind the same bot token.** The gateway `.env` filter (`curator-gateway.sh:72-123`) and `bootstrap-curator.sh:1046-1055` disable logic suggest correctness, but the env-unset only happens inside `curator-gateway.sh`; if an operator manually starts the gateway with onboarding also enabled and tokens in process env, both could poll. Falsifier: a Telegram `getUpdates` 409-conflict in practice. Not provable from source alone.
4. **Claim: operator gating is fail-closed on Discord guild channels.** Verified `_operator_discord_subject_allowed` returns False when no allowlist is set and `guild is not None` (`:182-188`). Falsifier: a DM path (`guild is None`) returning True with no allowlist (`:182-183`) — which it does, so a DM in the operator channel with no allowlist is *allowed*. That is intentional but is a wider surface than the code comment implies.
5. **Claim: the Telegram poison-pill loop cannot silently skip a good update.** `run_once` only advances offset after success or after hitting the failure limit (`:1186-1214`). Falsifier: if `record_onboarding_update_failure` itself throws, the `with` block raises out of `run_once` and the outer loop swallows it (`:1247-1252`) without advancing — an infinite retry on a DB-unwritable update. Plausible under disk-full.

## RISKS (severity-ranked, code-cited)
- **MEDIUM** — `settings`-table growth: every distinct Discord message id is persisted forever as `curator_discord_onboarding_seen_message:<id>` with no observed sweeper (`curator_discord_onboarding.py:194-204`). Long-lived high-traffic Discord onboarding bloats the `settings` table monotonically.
- **MEDIUM** — Dead/ misleading dismiss action: "Dismiss" upgrade/SSOT buttons write `arclink_upgrade_last_dismissed_sha` which nothing reads, yet tell the operator the notice was dismissed (`curator_onboarding.py:975-977`; `discord:272-273`). Operators may believe a notice is silenced when the next check still surfaces it (suppression keys off `_last_notified_sha`).

## VERDICT
This piece **provably does its job** as a transport+gating shell around CANON-04's onboarding engine. Load-bearing strengths, all code-verified: (1) the operator gate is genuinely unified with the hosted-API path via the shared `operator_telegram_sender_allowed` helper (seam 3), so the same Telegram update cannot get two different verdicts; (2) mutating operator commands are double-gated (allowlist + `hmac.compare_digest` approval code) and fail closed on exceptions (`:372`, `:840`); (3) component-callback ownership is checked against the stored session owner on every Notion/backup/ack path; (4) the Telegram poison-pill ledger advances the offset only on success or after a bounded retry limit, so the loop cannot wedge forever on one bad update; (5) the gateway/onboarding token de-confliction (`curator-gateway.sh` + `has_curator_non_onboarding_gateway_channels`) is a real interlock against double-binding. Real weaknesses: the `arclink_upgrade_last_dismissed_sha` write is dead and its user-facing copy is misleading (MEDIUM drift), the Discord seen-message dedupe leaks rows into `settings` with no sweeper (MEDIUM), and the prior ground-truth doc materially mischaracterizes this as a legacy/undocumented path when it is the live, systemd-managed curator surface with a 23-command operator overlay. None of these break the happy path; they are correctness/hygiene and doc-trust issues.
