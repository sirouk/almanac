# CANON-09 — Ingress & DNS

## PIECE
This piece is the **desired-vs-actual DNS / ingress record computation and persistence layer** for ArcPod deployments. It owns exactly one tracked file: `python/arclink_ingress.py` (284 lines, `git ls-files` confirms it is the sole member; no unlisted sibling exists). It is a thin, mostly-pure planning module that: (a) computes the *desired* Cloudflare DNS record set for a deployment given `prefix`/`base_domain`/`target`/`ingress_mode` (`desired_arclink_dns_records`, `desired_arclink_ingress_records`, ingress.py:35-60); (b) persists those desired records into the `arclink_dns_records` table with a status state machine (`persist_arclink_dns_records`, ingress.py:63-104); (c) renders Traefik dynamic router labels per host-role for domain vs tailscale-path ingress (`render_traefik_dynamic_labels`, ingress.py:235-284); and (d) provides DB-side teardown/status helpers (`arclink_dns_records_for_teardown`, `mark_arclink_dns_torn_down`, `_mark_dns_status`, ingress.py:137-204). It ALSO defines three `cloudflare`-object-driven convenience wrappers (`provision_arclink_dns`, `reconcile_arclink_dns`, `teardown_arclink_dns`) and a `DnsDrift` dataclass + `_parse_cloudflare_drift`, but those three wrappers are **not used by any production caller — only by `tests/test_arclink_ingress.py`** (see CODE vs DOC DRIFT and RISKS). The module holds NO secrets, opens NO sockets, runs NO subprocess; it only reads/writes one SQLite table, appends events, and returns dataclasses/dicts. The real Cloudflare HTTP calls and the live executor adapter live in CANON-11 (`arclink_executor.py`); config/env (`CLOUDFLARE_*`) lives in CANON-27; the apply/teardown orchestration that *calls* this module lives in CANON-08 (`arclink_provisioning.py`, `arclink_sovereign_worker.py`) and CANON-14 (`arclink_action_worker.py`).

## OUTPUT CONTRACT (code-verified)

Always `CNAME`, always `proxied=True`. Hostnames are `u-<prefix>.<domain>` (dashboard) and `hermes-<prefix>.<domain>` (hermes) per adapters.py:234,237.
- **DB write `arclink_dns_records`** (ingress.py:72-103): `INSERT ... VALUES (...,'desired',...) ON CONFLICT(record_id) DO UPDATE`. - **`_mark_dns_status` DB write** (ingress.py:145-148): `UPDATE arclink_dns_records SET status=?, updated_at=? WHERE deployment_id=?` — bulk-updates ALL rows for the deployment to one status; commits; then `append_arclink_event` (ingress.py:150-156).
- **Event side-effects** (CANON-01 `arclink_events`): `dns_drift` per drift item (`reconcile_arclink_dns`, ingress.py:127-133); `dns_teardown` (via `mark_arclink_dns_torn_down` -> `_mark_dns_status`, ingress.py:168); `dns_provisioned` (via `provision_arclink_dns`, ingress.py:225). Each is `append_arclink_event(conn, subject_kind="deployment", subject_id=deployment_id, event_type=..., metadata={...})` (ingress.py:127-133,150-156).
- **`arclink_dns_records_for_teardown` returns** `tuple[dict,...]` with keys `{"hostname","record_type","provider_record_id"}`, filtered `status != 'torn_down'`, ordered `hostname, record_type` (ingress.py:172-189).
- **`teardown_arclink_dns` returns** `list[str]` of removed hostnames from `cloudflare.teardown_records(...)` (ingress.py:202-204).
- **`provision_arclink_dns` returns** the `records` dict; on success marks status `provisioned` + emits `dns_provisioned` (ingress.py:225-227). On exhaustion of retries re-raises `last_error` (ingress.py:230-231).
- **`render_traefik_dynamic_labels` returns** `dict[role -> dict[label_key -> str]]`. Domain mode: `{"dashboard":{...host-rule labels...}, "hermes":{...}}` via `render_traefik_http_labels` (ingress.py:277-283). Tailscale-path mode: `{"dashboard":{}, "hermes":{**root_labels, **alias_labels}}` via `render_traefik_http_path_labels` (ingress.py:257-276) — note `dashboard` label dict is intentionally empty (ingress.py:258).

Drift: doc attribution wrong; code wins.
2. **Three "live" wrappers are test-only dead code in production.** `provision_arclink_dns` docstring "Create DNS records with retry safety. Idempotent via upsert." (ingress.py:217) and `teardown_arclink_dns` docstring (ingress.py:200) imply a live path, but `rg` shows zero production callers — only `tests/test_arclink_ingress.py:49,86,115,122,147,151`. The prior doc (line 243-244) lists `provision_arclink_dns`/`reconcile_arclink_dns` as if they are the operative DNS persistence path; they are not on the live path.
3. **Unused tailscale params.** `desired_arclink_ingress_records` accepts `tailscale_dns_name`/`tailscale_host_strategy` (ingress.py:50-53) but never reads them (tailscale branch returns `{}` at ingress.py:59). The names imply they influence output; they do not. INFO-level.
4. **`proxied` silently dropped on persist.** `DnsRecord.proxied` exists (adapters.py:184) and `persist_arclink_dns_records` is handed it, but it is not a column in `arclink_dns_records` and is never written (ingress.py:74-77). Name/docstring "retry safety" doesn't convey that it retries non-retryable errors too.

## ADVERSARIAL SELF-CHECK
1. **Claim: the three `cloudflare`-object wrappers have NO production caller.** Falsifier: a caller importing `provision_arclink_dns`/`reconcile_arclink_dns`/`teardown_arclink_dns` outside tests, or a dynamic `getattr`/`importlib` dispatch. I grepped `python/ bin/ tests/` (only tests hit). A reflective/string-based call would falsify; I did not exhaustively scan for `getattr(ingress, ...)`.
2. **Claim: ingress reads no env vars.** Falsifier: an indirect env read inside an imported adapter helper at module-import time. The imported functions (`arclink_hostnames`, etc.) are pure (adapters.py:228-374, no `os.environ`); `append_arclink_event`/`utc_now_iso` don't read env. I did not audit transitive imports of `arclink_control` for import-time env reads (out of this piece's scope).
3. **Claim: the `(LOWER(hostname),UPPER(record_type))` UNIQUE index can make `persist_arclink_dns_records` raise instead of upsert across deployments.** Falsifier: hostnames are globally unique because they embed a unique `prefix`, so two deployments never collide. This holds IF `prefix` is guaranteed unique per deployment (owned by CANON-08, not verified here). If a prefix were ever reused, the `ON CONFLICT(record_id)` clause would NOT catch a hostname-index collision and the INSERT would raise `IntegrityError`. I did not prove prefix uniqueness.
4. **Claim: `_mark_dns_status` bulk-updates every row for a deployment to one status.** Falsifier: a `WHERE` qualifier I missed — but ingress.py:146 has only `WHERE deployment_id = ?`. Confirmed. The risk is collateral: a partially-provisioned deployment teardown marks even still-`desired` rows `torn_down`.
5. **Claim: teardown's `provider_record_id` is populated by CANON-08, not ingress.** Falsifier: a hidden ingress writer. `rg provider_record_id python/arclink_ingress.py` shows only the SELECT/read (ingress.py:174,186), never an UPDATE/INSERT of it. Confirmed.

## RISKS (severity-ranked, code-cited)
- **MEDIUM — Dead/divergent DNS API surface.** `provision_arclink_dns`/`reconcile_arclink_dns`/`teardown_arclink_dns` (ingress.py:114,192,207) implement a `cloudflare`-object protocol that production never uses (production uses CANON-11 `executor.cloudflare_dns_apply`). They are green in tests but give false confidence that ingress owns the live Cloudflare write path. Maintenance/clarity risk and a doc-trust trap (prior doc treated them as the operative path).
- **MEDIUM — Bulk status clobber on teardown/mark.** `_mark_dns_status` UPDATEs ALL rows for a `deployment_id` to a single status with no per-record discrimination (ingress.py:145-148). A teardown after a partial apply will mark `desired`/never-provisioned rows as `torn_down`, masking that they were never created in Cloudflare.
- **LOW — `provider_record_id` correctness depends on an out-of-piece writer.** ingress' teardown read (ingress.py:174-186) expects `provider_record_id`, but ingress never writes it; if CANON-08 `_mark_dns_provisioned` backfill is skipped (e.g. apply skipped, tailscale, or a code change), teardown silently relies on the executor's find-by-name fallback (executor.py:2586). Cross-piece coupling that is invisible from this file.
- **LOW — `except Exception` retry breadth.** `provision_arclink_dns` retries on any exception, not just transient/network ones (ingress.py:228), so a deterministic failure (e.g. validation) is retried `1+max_retries` times before re-raising — wasted work, and the prior `last_error` masks earlier attempt context.
- **INFO — `persist_arclink_dns_records` commits per call (ingress.py:104) and `_mark_dns_status` commits mid-deployment (ingress.py:149).** If a caller wraps these inside its own transaction expecting atomicity, the internal `commit()` breaks that boundary. No caller currently relies on outer atomicity, but it is a latent footgun.
- **INFO — Unused params (`tailscale_dns_name`, `tailscale_host_strategy`) in `desired_arclink_ingress_records` (ingress.py:50-53).** No correctness impact; misleading signature.

## VERDICT
This piece **provably does its core job**: computing the desired domain-mode CNAME/proxied record set for exactly the two public host-roles (`u-<prefix>`, `hermes-<prefix>`), persisting them into `arclink_dns_records` with a `desired`->`provisioned`->`torn_down` status state machine, emitting the matching `arclink_events`, and rendering Traefik router labels for both domain (host-rule) and tailscale (PathPrefix+StripPrefix) ingress. Every live seam to CANON-08 (provisioning intent, persist, teardown-read), CANON-11 (executor request/result shapes), and CANON-01 (events/schema) is byte-for-byte consistent, both ends verified, and the 6 tests in `tests/test_arclink_ingress.py` exercise the persist/desired/preserve-provisioned logic and the golden Traefik labels. Load-bearing strengths: it is pure/side-effect-narrow (one table, no secrets, no network), the status-preservation upsert (ingress.py:82-89) correctly avoids demoting a live `provisioned` record on a no-op re-persist, and host-role narrowing (`ARCLINK_HOST_ROLES`) correctly prevents Files/Code from getting standalone subdomains (matching ingress-plan.md, contradicting the stale sovereign-control-node.md "four hosts" claim).
