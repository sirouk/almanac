# CANON-11 — Executor

## PIECE
The Executor is the single deploy/ops step engine for ArcLink: a fail-closed, dependency-injected boundary that turns a rendered "intent" mapping into real Docker Compose operations and provider mutations (Cloudflare DNS/Access, Chutes keys, Stripe actions, rollback). It owns exactly one tracked file, `python/arclink_executor.py` (2947 lines; `git ls-files` shows it is the only `*executor*` source under `python/`). It defines: the request/result dataclasses (`DockerComposeApplyRequest/Result`, `DockerComposeLifecycleRequest/Result`, `CloudflareDnsApplyRequest/Result`, `CloudflareDnsTeardownRequest/Result`, `CloudflareAccessApplyRequest/Result`, `ChutesKeyApplyRequest/Result`, `StripeActionApplyRequest/Result`, `RollbackApplyRequest/Result`, `DryRunStep`, `ResolvedSecretFile`, `ArcLinkExecutorConfig` `python/arclink_executor.py:159-360`); the injectable runner/client Protocols `DockerRunner`/`ChutesKeyClient`/`StripeActionClient` (`:362-444`); three real `DockerRunner` implementations (`SubprocessDockerComposeRunner`, `SshDockerComposeRunner`, `BrokeredDockerComposeRunner`) plus a recording `FakeDockerRunner` (`:447-738, 774-831`); two `SecretResolver`s (`FakeSecretResolver`, `FileMaterializingSecretResolver`) `:180-211`; the `ArcLinkExecutor` orchestrator class `:843-1765`; the `executor_for_fleet_host(...)` factory `:79-142`; and the path-validation / secret-materialization / digest / Cloudflare-HTTP / rollback-planning module functions. It does NOT own its callers (CANON-08 sovereign worker, CANON-13 pod migration, CANON-14 action worker) nor the broker server (CANON-12). Note the related test file `tests/test_arclink_executor.py` and steering docs `research/RALPHIE_EXECUTOR_*_STEERING.md` exist but are NOT in this piece's owned-file list.

## ADVERSARIAL SELF-CHECK
1. **Claim: production never sets `operation_conn`, so durable idempotency is inert.** Falsifier: a caller in a module I did not grep (e.g. a worker entrypoint) constructing `ArcLinkExecutor(operation_conn=conn)`. ## RISKS (severity-ranked, code-cited)
- **HIGH — Durable cross-process idempotency for live Chutes/Stripe is inert in production.** `operation_conn` defaults `None` `:852,859` and is set only in tests (`tests/test_arclink_executor.py:595,676`); all `_*_operation_idempotency` helpers no-op on `None` `:2708,2730,2755,2781`. Live `chutes_key_apply`/`stripe_action_apply` then rely solely on the process-local `_live_chutes_runs`/`_live_stripe_runs` dicts `:867-868`, which do NOT survive a restart — a retried Stripe refund/cancel after a crash could double-execute against the provider. Provider-side `idempotency_key` is still passed `:1248,1368`, which mitigates IF the provider honors it, but the ArcLink ledger does not record it.
- **MEDIUM — Live SSH/local executor is root-equivalent on the worker.** `SubprocessDockerComposeRunner`/`SshDockerComposeRunner` shell out to `docker compose`/`docker run --user root` `:503,605,2384` (GAP-019 trusted-host residual). Containment rests on path/allowlist validation (`_validate_deployment_config_paths` `:1817`, `_require_allowed_ssh_host` `:2153`) and `shlex.quote`; any gap in those is host-root.
- **MEDIUM — `allow_lifecycle_project_override` weakens the project==`arclink-{id}` invariant.** When `ARCLINK_ACTION_WORKER_ALLOW_LIFECYCLE_PATH_OVERRIDES` is truthy `:93`, `_validated_docker_compose_lifecycle_paths` passes `require_expected=False` `:1879`, allowing a caller-supplied project name that need not match the deployment id (still constrained to `^arclink-...`). Lifecycle (incl. teardown) could target a different `arclink-*` project than the deployment id implies.
- **LOW — Broker request sends `remove_volumes`/`include_all` for operations that ignore them.** `:802-803` always populates both; server only honors them per-operation `:111-113`. No security impact, but a future operation reusing these keys could silently change behavior.
- **LOW — `StrictHostKeyChecking=accept-new` default for SSH runner.** `:122,529` accepts new host keys on first contact (TOFU); first-connection MITM is possible unless `ARCLINK_FLEET_SSH_KNOWN_HOSTS_FILE` is pinned `:128-129`.

The Executor provably does its core job: it is a genuinely fail-closed (`_require_live_enabled` on every mutator except dry-run), dependency-injected step engine with strong, code-enforced path containment (deployment-scoped root, env==`arclink.env`, compose==`compose.yaml`, project==`arclink-{id}`), atomic 0600 secret/file materialization with flock, exception-time secret cleanup, topological service ordering with cycle detection, secret-redacted error surfaces, and a clean three-way runner seam (subprocess/ssh/brokered) whose brokered wire contract is verified byte-for-byte against the CANON-12 server. Load-bearing strengths: the broker double-validation and the volume-preservation-by-default teardown contract are real and correct. Real weaknesses: (1) the durable `arclink_operation_idempotency` ledger is wired but dead in production because no caller injects `operation_conn`, leaving live Chutes/Stripe idempotency to a volatile in-process dict (HIGH); (2) the entire `rollback_apply` live API has no production caller; (3) the live Docker path is trusted-host root-equivalent (GAP-019) and the lifecycle project-override env relaxes a key invariant. The piece is well-built but ships a richer live API than production actually exercises, and its strongest idempotency guarantee is the one production silently disables.
