# CANON-14 — Operator & Admin Control

## PIECE
This piece is ArcLink's operator/admin control surface: the chat-native Operator Raven command layer, the operator's single in-stack Hermes identity, the action worker that drains queued admin-action intents, the rollout model, and the admin CLI. It owns exactly five tracked files:

- `python/arclink_ctl.py` (2967 lines) — the operator/admin CLI (wrapped by `bin/arclink-ctl`). Domain/action argparse dispatch for tokens, requests, onboarding, SSOT, agents, vault, channel reconfigure, user purge/deenroll, upgrade-check, retrieval replay, org-profile, internal curator-refresh/pin-upgrade-check, notion identity/webhook ops, notifications, provision requests. It is the *origination* point for the operator-upgrade detector (`upgrade_check`, line 1681) that queues operator+user notifications and writes `arclink_upgrade_*` settings.
- `python/arclink_operator_raven.py` (2424 lines) — the operator chat command surface. Read commands never mutate; the seven mutating commands (`pod_repair`, `rollout`, `host_upgrade`, `pin_upgrade`, `upgrade_sweep`, `fleet_drain`, `fleet_resume`) follow a dry-run / fail-closed-actorless / require-confirm / queue contract. It QUEUES audited intents into two distinct queues (`arclink_action_intents` via the dashboard, `operator_actions` via control) plus mints one-tap upgrade button nonces. It never runs Docker/SSH/provider commands inline.
- `python/arclink_operator_agent.py` (467 lines) — the operator's single Hermes identity (control-DB only). Ensures a comped `operator` user + one `operator` deployment row stamped `operator_agent=True`, enforces a one-agent invariant, and bridges free-form operator chat to the in-stack Hermes gateway via the `public-agent-turn` notification worker.
- `python/arclink_action_worker.py` (2423 lines) — consumes `arclink_action_intents` rows (the queue `pod_repair`/`rollout` write to), claims atomically, dispatches by `action_type` to the executor (or local control-plane), records attempts/events/audit, recovers stale running actions, and GCs pod migrations. Also owns the Academy apply/post-apply-refresh materialization lane.
- `python/arclink_rollout.py` (1458 lines) — the legacy generic rollout model (`arclink_rollouts` waves) plus the GAP-032 ArcPod-update rollout: a pure dry-run planner, a local typed-rows materializer, and a record-only fake/local batch executor. No real per-Pod refresh exists.

All listed files exist and are tracked. No additional clearly-belonging file was found unlisted; adjacent producers/consumers (`arclink_control.py`, `arclink_dashboard.py`, `arclink_telegram.py`, `arclink_enrollment_provisioner.py`, `arclink_fleet.py`) are owned by other CANON pieces and treated here as boundary seams.

## ADVERSARIAL SELF-CHECK
1. **Claim: the approval code is the only second factor for mutating Raven commands.** Falsifier: a transport other than Telegram that calls `dispatch_operator_raven_command` directly with a forged `actor_id` and `--confirm` would bypass the approval-code wall entirely — the code check lives in `arclink_telegram.py:1305-1322`, NOT in Raven. Raven only requires `confirmed` (the literal token) + non-empty `actor_id`. I verified Telegram enforces the code; I did NOT audit every caller of `dispatch_operator_raven_command` (Discord/curator surfaces) for equivalent enforcement. This is the load-bearing security uncertainty.
2. **Claim: `mutation_performed` is a reliable "real change happened" signal.** Falsifier: for idempotent replays `mutation_performed=not already_queued`/`bool(created)`, so a second identical confirmed command returns `mutation_performed=False` even though the action exists. Verified in code (`:1150,1576,1652`); correct but subtle — callers must not treat False as "nothing queued".
3. **Claim: the action worker never runs live shell inline.** Falsifier: `academy_apply` writes real files (SOUL.md, vault md, state json) to the deployment's hermes_home when `writes_enabled`+`ARCLINK_ACADEMY_APPLY_LIVE` (`worker:2089-2206`). That is a real filesystem mutation, not Docker/SSH, but it IS a live write the docstring's "queues and resolves" framing for the *operator agent* does not cover (different module). Verified; flagged as MEDIUM risk below.
4. **Claim: `_claim_next_queued_action` is race-free.** Falsifier: it relies on SQLite `BEGIN IMMEDIATE` + a rowcount-guarded UPDATE. Under WAL with multiple workers this is correct, but I did not test concurrent workers; the guard (`cursor.rowcount != 1 → return None`) is the proof (`:484`). Confidence high but untested under real concurrency here.
5. **Claim: button nonces are single-use and unforgeable.** Falsifier: the nonce digest is `sha256(token_urlsafe(18))` stored in `settings`; `consume_operator_button_nonce` checks `used_at`/`expires_at` and rewrites `used_at` (`:1388-1394`) but the write is not transactionally guarded against a double-consume race within the TTL. Two simultaneous taps could both read `used_at==""` before either writes. Verified the logic; the race window is real but small and the mapped command is itself idempotent at the queue layer.

## RISKS (severity-ranked, code-cited)
- **MEDIUM** — Approval-code enforcement lives in the transport, not in Raven. `dispatch_operator_raven_command` requires only `confirmed` + non-empty `actor_id` (`arclink_operator_raven.py:412-420,400-409`); the code wall is `arclink_telegram.py:1305-1322`. Any other caller that supplies a forged/loose `actor_id` and the literal `confirm` queues real mutations. Defense-in-depth at the provisioner (`request_source=="operator-raven"`, `provisioner:2295-2297`) limits *which* queue executes, but the intent is still created.
- **MEDIUM** — Dismissed pin upgrades remain queueable. `_pending_pin_payloads_for_component`/`_pending_pin_payloads` (`arclink_operator_raven.py:1281-1290`) call `list_pin_upgrade_action_payloads(active_only=True)`, which filters only `applied_at IS NULL` and ignores `silenced` (DISSECT P1/M3, `control.py:9596-9608`). A dismissed release can still be re-queued via `/pin_upgrade` or `/upgrade_sweep`.
- **MEDIUM** — `academy_apply` performs live private-filesystem writes inside the action worker. `_materialize_academy_apply` writes `SOUL.md`/vault/state files when `writes_enabled`+`ARCLINK_ACADEMY_APPLY_LIVE` (`arclink_action_worker.py:2089-2206`). Path-traversal is guarded (`_safe_child_path` `:1351`, `_academy_safe_relative_path` `:1414`), but this is a real on-disk mutation gated by one env var, not the executor adapter.
- **LOW** — Button-nonce consume is not atomically guarded against a concurrent double-tap within the TTL (`arclink_operator_raven.py:1388-1394`). Impact bounded by queue-layer idempotency.
- **LOW** — `arclink_ctl channel reconfigure operator` and several flows read/handle bot tokens (`getpass`, env-file writes `arclink_ctl.py:2376-2627`); tokens are written to the operator env file at the configured path. Correctly 0600-guarded for password files, but token persistence to env files is broad attack surface (owned jointly with CANON-27).
- **INFO** — Prior ground-truth doc `research/ground-truth/05-operator-raven-control.md` is cite-stale (line numbers and the 4-vs-7 mutating set); should be superseded by this CANON-14 record.
- **INFO** — `_handle_status` `_safe_call` default dict omits `active_hosts`/`total_hosts` keys it later reads (`arclink_operator_raven.py:447,467`); harmless due to `.get(...or 0)`.

## VERDICT
This piece provably does its job: it is a fail-closed, identity-gated, audited operator/admin control plane. Load-bearing strengths, all code-verified end-to-end: (1) a clean dry-run / fail-closed-actorless / require-confirm / queue contract for the seven mutating Raven commands, with the actual mutation delegated to two well-defined queues and never run inline; (2) an atomic claim (`BEGIN IMMEDIATE` + rowcount guard) and honest status mapping in the action worker, with stale-action recovery and secret-scrubbed outputs; (3) a GAP-032 rollout skeleton that is genuinely pure-dry-run → local-typed-rows → record-only-batch with `live_mutation_performed=False` everywhere and an explicit fake/local-only executor contract; (4) a one-agent invariant for the operator's Hermes identity; (5) defense-in-depth on the host-upgrade/pin-upgrade lane (Telegram approval code → Raven confirm token → provisioner `request_source=="operator-raven"` re-check). Real weaknesses: the approval-code second factor is enforced by the Telegram transport rather than by Raven itself (so non-Telegram callers are the audit gap), dismissed pin upgrades remain queueable because the active filter ignores `silenced`, and `academy_apply` is a live filesystem write gated by a single env var. None of these break the core contract; they are the precise places an independent cross-check should focus.
