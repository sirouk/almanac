# CANON-21 — Org Profile

## PIECE
This piece is ArcLink's **operator-authored operating-context model**: a single private YAML profile that declares who the humans are, what they are responsible for, how teams/roles relate, the shared agent baseline/lineage, authority/approval rails, and operational workflows/automations/benchmarks — then **fans that one source out** into control-plane rows, a sanitized vault Markdown render, managed-memory sections, a per-agent context slice, a SOUL overlay, and an agent identity overlay JSON. It owns exactly five tracked files: `python/arclink_org_profile.py` (2208 lines — the model, JSON-schema + semantic + secret validation, all renderers, the SQLite mirror, and `apply_profile`/`doctor_profile`), `python/arclink_org_profile_builder.py` (723 lines — an interactive stdin/stdout TUI that builds/edits the YAML and shells out to `arclink-ctl org-profile apply`), `config/org-profile.schema.json` (757 lines — JSON Schema draft 2020-12), and two example profiles `config/org-profile.example.yaml` (582 lines) and `config/org-profile.ultimate.example.yaml` (1533 lines). Two clearly-belonging-but-unlisted files also exist and are noted but NOT owned here: `bin/org-profile-builder.sh` (thin PYTHONPATH wrapper; belongs to CANON-31's bin inventory) and `docs/org-profile.md` (CANON-32 docs) and `tests/test_arclink_org_profile.py` (CANON-29 test corpus).

## ADVERSARIAL SELF-CHECK
1. **"org_profile_* tables are truly never read."** Falsifier: a dynamic/string-built query, an external service, or a test reading them. I ran `rg "FROM org_profile_(people|roles|teams|relationships|revisions)"` → 0 hits in `python/`, but did not grep `bin/`, `web/`, or non-`FROM` access patterns (e.g. `pragma`, ORM). Medium confidence.
2. **"apply is fully fail-closed on secrets."** Falsifier: a secret shape not in `SECRET_VALUE_PATTERNS` and not in a secret-named key passes through (e.g. a bare high-entropy password in a field named `passphrase` — `passphrase` is not in `SECRET_KEY_TERMS`, and a random base64 blob matches no value pattern). I verified gh-token detection works even in `notes`, but the scanner is allowlist-shaped and will miss novel/unnamed secrets. Medium-high confidence the gap exists.
3. **"`_match_person_for_agent` precedence is person_id→unix_user→unique-display."** Falsifier: empty `org_profile_person_id` plus a unix_user that matches a DIFFERENT person than display would. Verified order in code (`:1347-1374`); unix_user wins over display, and display only matches when exactly one candidate. High confidence.
4. **"Both example YAMLs validate."** Verified by executing `validate_profile` → both `valid:True, 0 errors, 0 warnings`. But validation used `cfg=None`, so reference-path and seed-sha256 checks were skipped (they require `cfg`). Under a real `cfg` the `ultimate` example's `references`/`seed_sources` paths could warn. Medium confidence on the no-cfg result; low on the with-cfg result.
5. **"merge_soul_overlay is idempotent across re-applies."** Falsifier: if an operator hand-edits inside the markers, the next merge replaces the block (intended) but could strip manual edits silently. Verified marker-replace logic (`:1705-1707`); this is by-design data loss inside the managed block, which a reviewer might consider a footgun.

## RISKS (severity-ranked, code-cited)
- **MEDIUM — Write-only SQLite mirror.** `_replace_profile_rows` populates 5 tables (`arclink_org_profile.py:1961-2044`) that no reader consumes (no `SELECT FROM org_profile_*` in tree). Dead state that can drift from `applied.json` and wastes a full DELETE+INSERT per apply; misleads future maintainers into trusting rows that are never read. (`:1961`)
- **MEDIUM — No concurrency control on apply.** DELETE→INSERT→commit (`:1965-2053`) plus multi-file fan-out has no lock; a concurrent apply or a managed-context refresh reading `applied.json` mid-write can observe inconsistent revision vs slices. Atomic per-file writes do not make the *set* of writes atomic. (`:2093-2145`)
- **MEDIUM — Allowlist secret scanner.** `_secret_scan_errors` only catches secret-named keys (`:71`) or known value regexes (`:84`); a high-entropy secret in a benign-named field (e.g. `passphrase`, `notes` with a non-matching format) passes validation and is persisted to `applied.json` and SQLite. Fail-open for unknown secret shapes. (`:233`)
- **LOW — `cpk_` placeholder bypass.** `_is_placeholder_secret` treats any `cpk_...` value as a placeholder and skips scanning it (`:216`). A real Chutes public key (or anything an attacker prefixes with `cpk_`) is exempted from secret detection; benign for public keys, but the prefix-based exemption is unverified-content. (`:216`)
- **LOW — Unmatched-agent slice deletion is silent data action.** On apply, any active user agent that no longer matches loses its context slice via `stale_path.unlink()` (`:2123`) with only a report flag; no audit/event emitted from this module. (`:2122`)
- **LOW — Scope limited to `role='user'` active agents.** `_active_agent_rows` (`:2069`) excludes non-user/non-active agents from context fan-out; if operator/curator agents need org-profile orientation they get none from `apply_profile`. (`:2069`)
- **INFO — Dual `default_profile_path` with hardcoded fallback.** Builder fallback hardcodes `arclink-priv/config` (`builder.py:45`); diverges from `cfg.private_dir` if that dir is named differently. (`builder.py:37`)

## VERDICT
**Yes — this piece provably does its core job.** Executed validation confirms: both example profiles and the builder starter validate cleanly; schema+semantic+secret layers all fire (verified a real GitHub token is rejected even in a non-secret-named field); `apply_profile` is genuinely **fail-closed** (returns without side effects on invalid input, `:2084`); every output is atomic, idempotent, and mode-restricted (`0o600` for secret-bearing `applied.json`/identity/SOUL, `0o644` for sanitized vault/slices). The fan-out seams to CANON-01 (`settings`), CANON-04/08 (`agent_identity.org_profile_person_id`), and CANON-19 (managed-context payload + `materialize_agent_context`) are all **both-ends key-verified** — producer keys match consumer reads exactly. Load-bearing strengths: single-source-of-truth design, sanitizing renderer that omits restricted fields by default (`:1015`,`:1063`), checksum-as-revision discipline (`:181`), and a privacy-aware people-detail gate (`_vault_render_allows_people_details` `:807`). Real weaknesses: the SQLite mirror is **write-only dead state** (the authoritative read path is the JSON file, not the rows it laboriously maintains); there is **no lock** around the multi-write apply; and the secret scanner is **allowlist/best-effort**, not a guarantee. None of these break the happy path, but the dead-table mirror and the "secrets cannot enter" overclaim are the two items most worth an independent Codex pass.
