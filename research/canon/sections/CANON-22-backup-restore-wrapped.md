# CANON-22 — Backup / Restore / Lifecycle / Wrapped

## PIECE

This piece owns ArcLink's two git-backup lanes, the local restore-smoke verifier, and the "ArcLink Wrapped" year-in-review report engine. The exact tracked files are: `python/arclink_wrapped.py` (1179 lines — Wrapped scoring/render/persist/cadence/enqueue/scheduler/admin), `bin/backup-agent-home.sh` (per-user Hermes-home → private GitHub backup), `bin/configure-agent-backup.sh` (two-phase pending→verify activation for the Hermes-home lane), `bin/backup-to-github.sh` (control-plane `arclink-priv` → private GitHub backup), `bin/arclink-restore-smoke.sh` (local-only artifact-shape restore verifier; also a CANON-28 CI member), and `bin/arclink-wrapped.sh` (13-line wrapper that execs `arclink_wrapped.py`). One additional clearly-belonging tracked file exists but is owned by CANON-29: `tests/test_arclink_wrapped.py`. There is NO `systemd/` file in this piece — the systemd timer that drives `backup-to-github.sh` (`arclink-github-backup.service`/`.timer`) lives in CANON-26, and the per-agent backup is driven by a Hermes cron job installed by `bin/install-agent-cron-jobs.sh` (CANON-26), not a tracked timer. The Wrapped Docker job-loop service is declared in CANON-08 (`python/arclink_provisioning.py`).

## ADVERSARIAL SELF-CHECK

1. **"captain-wrapped extra_json keys exactly match consumer reads."** I verified producer keys (`:912-920`) and consumer reads `extra.user_id`/`extra.report_id`/`row.message`/`row.channel_kind`/`row.target_id`. Falsifier: if delivery code elsewhere reads `extra.period` or `extra.novelty_score` and breaks when absent — but those are present, so low risk. The `render_kind` key is written but I did not confirm a consumer reads it (it may be dead metadata).
2. **"Wrapped never deletes/over-writes a generated report."** `_record_wrapped_failure` uses `INSERT OR REPLACE` (`:980`) on a *failure* report_id that includes the attempt count (`:138`), so it cannot clobber a `generated` row (different primary key). Falsifier: a hash collision in `_failure_report_id` sanitization producing the same id as a real `wrap_*` report — implausible (prefix `wrap_failed_`).
3. **"backup-to-github excludes the deploy key from the commit."** Verified the exclusion loop (`:62-76`) only fires when the key path `path_is_within_dir $ARCLINK_PRIV_DIR`. Falsifier: if the default key path (`$ARCLINK_HOME/.ssh/...`, `common.sh:320`) is *outside* priv dir, exclusion is a no-op — but then the key is not in the tree at all, so it is still safe. Edge: a misconfigured `BACKUP_GIT_DEPLOY_KEY_PATH` placed inside the vault but symlinked could evade `realpath`-based containment; not tested.
4. **"restore-smoke cannot write outside restore-dir."** tar member validation (`:104-114`) rejects `/*`, `../*`, `*/../*`; but it only runs for `*.tar/.tgz` files (`:132`), NOT for the `git archive HEAD` (`:119`) or directory-snapshot (`:125`) paths, which trust git/tar to stay relative. Falsifier: a directory source containing a symlink that tar follows to an absolute target — tar `-cf .` stores symlinks as symlinks by default, so extraction recreates the symlink in restore-dir, not outside; low risk but unproven against crafted symlinks.
5. **"`_has_wrapped_signal` truly blocks empty reports."** It requires ≥1 nonzero among events/audits/pod_messages/memory_cards (`:404-421`) and returns False on empty `deployment_ids`. Falsifier: a Captain with a session_counter turn count but zero of the four DB signals would be skipped despite real Hermes activity — `session_count`/`turn_count` are NOT in the signal gate (`_wrapped_signal_counts:365-401`). This is a real coverage gap (see Risks).

## RISKS (severity-ranked, code-cited)

- **MEDIUM — Hermes-session activity bypasses the Wrapped eligibility gate.** `_has_wrapped_signal` checks only events/audits/pod_messages/memory_cards (`arclink_wrapped.py:365-401,404-421`); a Captain whose only activity is Hermes turns/sessions (injected `session_counter`) is treated as "no signal" and never gets a `missing` report generated. Real activity → no Wrapped. `arclink_wrapped.py:1095`.
- **MEDIUM — No backoff on failed-report retry.** `list_due_wrapped_captains` re-queues every `failed` period each scheduler tick with no time gate (`arclink_wrapped.py:1091-1094`); the only brake is the 300s loop and the attempt-3 operator notice. A permanently-failing Captain generates a new failure row + churn every cycle. `arclink_wrapped.py:1091`.
- **MEDIUM — Visibility check is fail-open on transient GitHub errors only by refusal, but 404 maps to "non-public-or-missing" and is allowed.** `backup-agent-home.sh:88-90` and `configure-agent-backup.sh:165-167` treat HTTP 404 as `non-public-or-missing` and proceed with backup (only `public` and `error:*` block). A repo that returns 404 because the deploy key lacks read scope is indistinguishable from a private repo, so a misconfigured/wrong repo is not blocked at the visibility gate. `backup-agent-home.sh:181-188`.
- **LOW — tar path validation only covers archive sources.** restore-smoke validates member paths for `*.tar/.tgz` (`arclink-restore-smoke.sh:104-114,133`) but not for `git archive` or directory-snapshot extraction, trusting tar/git to stay relative. Crafted symlinks in a directory source are not explicitly screened. `arclink-restore-smoke.sh:116-128`.
- **LOW — Duplicated reconcile logic across two scripts can drift.** `reconcile_backup_git_remote_branch` is copied verbatim into `backup-to-github.sh:13-54` and `backup-agent-home.sh:101-142`; a future fix to one will silently miss the other. `backup-agent-home.sh:101`.
- **INFO — `force-with-lease` unrelated-history alignment depends on an unenforced single-writer assumption.** Comment at `backup-to-github.sh:136-137` asserts single-writer; no lock guards concurrent runs. `backup-to-github.sh:139-141`.
