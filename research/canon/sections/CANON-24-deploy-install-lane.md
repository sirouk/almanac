## CODE vs COMMENT/DOC/NAME DRIFT
- **AGENTS.md 12-step `./deploy.sh upgrade` (`AGENTS.md:182-195`) describes `run_root_upgrade`, but `./deploy.sh upgrade` does NOT call it.** The live dispatch rewrites `upgrade`→control and runs `run_control_install_flow 0` (Docker Compose rebuild) (`bin/deploy.sh:13012-13016`,`:12835-12837`,`:11589`). `run_root_upgrade` (the rsync-public-repo + bootstrap-system + userland + curator + strict-health flow the doc lists) is reachable ONLY via the privileged `--apply-upgrade` flag (`:12998`), which is the RETIRED shared-host lane (`run_upgrade_flow` at `:8224` is only invoked from retired paths). The doc's numbered steps are accurate for the retired worker, stale for the public command.
- **AGENTS.md:114 "`deploy.sh` is a thin wrapper around `bin/deploy.sh`"** — TRUE (`deploy.sh:5`). No drift.
- **`bin/install-deployment-hermes-home.sh` default `HERMES_HOME` is `/home/arclink/.hermes`** (`:5`) but AGENTS.md:564 implies it is the ArcPod home; the script is generic and the caller passes the real home — name/comment is fine, just note the bare default differs from runtime usage.
- **component-upgrade.sh header comment (`:1-33`) says apply "rewrites pins.json + commits + pushes + re-execs `./deploy.sh upgrade`"** — but with `--skip-upgrade` (the host-runner's actual invocation) it does NOT re-exec (`:664`). The header describes the default, not the host-runner path; mild drift between comment and the dominant real caller.
- **`init.sh:13-14` default `REPO_URL=https://github.com/example/arclink.git` / `RAW_INIT_URL=.../example/arclink/main/init.sh`** are placeholder `example` URLs; if `LOCAL_REPO_DIR` is unset and these defaults stand, `ensure_repo_cache` clones a non-existent repo (`init.sh:117-129`). ## RISKS (severity-ranked, code-cited)
- **[HIGH] Host-runner-triggered pin upgrades push to the live upstream branch.** `bin/component-upgrade.sh:658-662` commits+pushes `config/pins.json` to `ARCLINK_UPSTREAM_BRANCH` whenever `--skip-push` is absent, and the host-runner invokes apply with only `--skip-upgrade` (`arclink_operator_upgrade_host_runner.py:276`). A Raven/operator queued pin bump therefore mutates the production git branch automatically (requires the deploy key to be present; `require_upstream_push_ready` fatals otherwise, `:372-376`).
- **[MEDIUM] Documentation drift makes the upgrade flow's real behavior non-obvious.** `AGENTS.md:182-195` describes a flow (`run_root_upgrade`) that the public `./deploy.sh upgrade` no longer runs (`bin/deploy.sh:13012-13016`). An operator reasoning from the doc will expect rsync/bootstrap/strict-health, but get a Docker rebuild.
- **[MEDIUM] Placeholder upstream URLs in entry scripts.** `init.sh:13-14` defaults to `github.com/example/arclink.git`; `bin/bootstrap-userland.sh:17` defaults to a personal fork `github.com/sirouk/arclink.git`. A bootstrap that relies on the default clones the wrong/nonexistent repo (fail) or a fork (silent drift).
- **[MEDIUM] docker-entrypoint secret repair fails open on unwritable config.** When `config_file_can_write`/`config_file_can_repair` are false the entrypoint only `echo Warning` and continues (`:655-660`,`:696-698`); a pod can start with `change-me`/empty secrets if no split mount supplies them.
- **[LOW] `run_control_install_flow` array expansion under `set -u`.** `for arg in "${CONTROL_DEPLOY_ARGS[@]:-}"` (`bin/deploy.sh:11599`) iterates an empty-string element when no args, harmless here but a latent footgun if `arg` were used unguarded.
- **[LOW] `init.sh:199` requester identity fallback uses `printf "$TARGET_USER"` as a format string** — if a username contained `%` it would be interpreted; low impact (usernames rarely contain `%`).
- **[INFO] `bin/deploy.sh` stable-copy uses `mktemp` in `$TMPDIR`/`/tmp`** (`:12`) world-readable parent; the copy is 0700 so contents are protected, but the existence/timing is observable.

## VERDICT
The piece provably does its job for the LIVE lane: `./deploy.sh {install,upgrade,health}` deterministically routes to the Dockerized Control Node flow (`run_control_install_flow`), which is dirty-tree-guarded, ff-only-synced, and installs the operator-upgrade host-runner timer — and the P7 host-runner↔component-upgrade.sh seam is tight (allowlisted components/flags on both ends, machine-readable status markers re-parsed by the producer, both ends verified in code). The Docker image build correctly reads pinned versions from `config/pins.json`, and the entrypoint idempotently materializes runtime config + secrets with atomic writes and 600 perms. Load-bearing strengths: the self-reexec stable-copy (upgrade can't corrupt the running script), idempotent secret repair, and the allowlist-only host-runner argv. Real weaknesses: (1) a HIGH that operator-queued pin bumps auto-push to the production branch; (2) the dominant operator-facing doc (`AGENTS.md`) describes a retired worker flow, so the actual upgrade behavior is mis-documented; (3) placeholder/personal-fork upstream URLs in the entry scripts that only work because real deployments override them; (4) several fail-open warnings in the entrypoint that can let a pod boot with placeholder secrets. The retired shared-host install/upgrade functions (`run_root_install`/`run_root_upgrade`/`run_upgrade_flow`) remain fully present and reachable via `--apply-*`, which is dead weight and a source of the doc drift.
