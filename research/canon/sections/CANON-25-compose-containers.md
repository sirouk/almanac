# CANON-25 — Container Topology (Compose)

## PIECE
This piece owns the **Control Node container substrate** — the declarative Docker Compose topology and the shell helpers that drive it. It owns exactly 8 tracked files (all present, none missing):
- `compose.yaml` (1178 lines) — the single Compose project `name: arclink` (`compose.yaml:1`) declaring **51 services**, **1 named volume** (`arclink-qmd`, `compose.yaml:1158-1159`), and **9 networks** (`compose.yaml:1161-1177`). Defines five YAML anchors (`x-arclink-app`, `x-arclink-env`, `x-arclink-control-secret-env`, `x-arclink-operator-env`, `x-arclink-job`, `x-nextcloud-db-env`) plus three healthcheck/restart anchors.
- `compose/nextcloud-compose.yml` (49 lines) — a *standalone* 3-service (`db`/`redis`/`app`) Nextcloud Compose file used by `bin/nextcloud-up.sh`/`down.sh`, distinct from the `nextcloud`/`postgres`/`redis` services embedded in `compose.yaml`.
- `bin/arclink-docker.sh` (3478 lines) — the orchestrator: wraps `docker compose -f compose.yaml`, writes the generated `docker.env`, generates per-service secrets, runs `up`/`down`/`ps`/`ports`/`health`/`logs`/`config`/`build`/`teardown`, and a large set of Docker reconcile/enrollment/operator helpers.
- `bin/docker-job-loop.sh` (144 lines) — the recurring-job supervisor wrapping every `command:` that needs a polling loop; writes redacted per-job status JSON.
- `bin/docker-health.sh` (481 lines) — the in-container health probe (`ARCLINK_HEALTH_WATCH_HEALTH_CMD` for `health-watch`).
- `bin/docker-agent-supervisor.sh` (6 lines) — trivial `exec python3 arclink_docker_agent_supervisor.py` shim (the agent-supervisor `command:`).
- `bin/nextcloud-up.sh` / `bin/nextcloud-down.sh` — the standalone Nextcloud sidecar lifecycle (podman pod or compose; not used by the main `compose.yaml` stack, which has its own `nextcloud` service).

A clearly-belonging-but-not-listed seam helper is `compose()` / `bootstrap()` inside `bin/arclink-docker.sh:117-124,219` and the compose helpers in `bin/common.sh` (`run_compose`, `with_nextcloud_compose_env`, `have_compose_runtime`); `bin/common.sh` is CANON-24's file, so I cite but do not claim it.

## CODE vs COMMENT/DOC/NAME DRIFT
1. **`docs/docker.md:46` says control-ingress "no Docker socket"** — TRUE in code (`compose.yaml:613-644` has no socket mount, `cap_drop: ALL`). No drift; confirmed.
2. **`docs/docker.md:84-92` port/socket/root table** vs code: every row matches (`gateway-exec-broker 8911 write no`, `deployment-exec-broker 8912 write no`, `agent-supervisor-broker 8913 write no`, `migration-capture-helper 8914 none yes`, `agent-user-helper 8915 none yes`, `agent-process-helper 8916 none yes`, `operator-upgrade-broker 8917 none yes`). Confirmed against `compose.yaml:657,832,666,687,826,896,898,918,931,847,865`. No drift.
3. **DRIFT (key-name)**: `bin/docker-job-loop.sh:82` writes `"job"`; `bin/docker-health.sh:250` reads `data.get("job_name") or data.get("job")`. The status writer never emits `job_name` or `exit_code` (it writes `returncode`, `:84`); `docker-health.sh:252` reads `exit_code` first. The names diverge; only the `or`-fallbacks keep it working. A future health-check that drops the fallback would silently mis-parse every job.
4. **DRIFT (Nextcloud config split)**: `compose/nextcloud-compose.yml:40` hard-codes `OVERWRITEPROTOCOL: https` and mounts `${NEXTCLOUD_DATA_DIR}:/var/www/html/data` (`:44`); the embedded `nextcloud` service in `compose.yaml:297` uses `OVERWRITEPROTOCOL: ${NEXTCLOUD_OVERWRITEPROTOCOL:-http}` (default **http**) and mounts `./arclink-priv/vault:/srv/vault` (`:305`) with **no** explicit data-dir mount. Two Nextcloud topologies with divergent protocol defaults coexist; `docs/docker.md` does not mention the standalone `compose/nextcloud-compose.yml`.
5. **DRIFT (comment vs literal)**: `compose.yaml:844-847` comment says the broker "must run as root inside the container when the live checkout is under /root". The `user: "0:0"` is unconditional, not gated on the checkout path — the name/comment implies conditionality that the YAML cannot express. Code (the static `0:0`) wins.
6. **NAME drift**: `compose.yaml:602-611` `control-web` declares a healthcheck on `:3000` but publishes **no** host port; the doc (`docs/docker.md:45`) calls it the web surface, but host traffic actually reaches it via `control-ingress` (`:628-630`), not control-web directly.
7. **`docs/docker.md:75` "All seven high-authority services ... refuses to start unless ...=accepted"** — CONFIRMED in code (seam #5), and the default is empty so they do refuse. The doc is accurate; flagged here only because the consequence (default = non-booting brokers) is non-obvious.

## RISKS (severity-ranked, code-cited)
- **HIGH — 3 writable Docker-socket holders are root-equivalent (acknowledged GAP-019).** `deployment-exec-broker` (`compose.yaml:666`), `agent-supervisor-broker` (`:832`), `gateway-exec-broker` (`:1017`) each mount `/var/run/docker.sock` rw. A compromise of any equals host root. Mitigation is tokens + scoped internal nets + `cap_drop: ALL`, not isolation. Matches `docs/docker.md:94-98`.
- **HIGH — 4 services run `user: "0:0"` (root in container).** `migration-capture-helper` (`:679`), `operator-upgrade-broker` (`:847`, `cap_add: DAC_OVERRIDE` `:851`), `agent-user-helper` (`:885`, `cap_add CHOWN/DAC_OVERRIDE/FOWNER`), `agent-process-helper` (`:918`). They drop the socket but root + writable private-state/host-repo binds remain high-authority.
- **MEDIUM — operator-upgrade-broker mounts the live host repo writable as root** (`compose.yaml:869`, no `:ro`). Unlike `agent-supervisor`/`curator-refresh`/`agent-process-helper` which mount the repo `:ro` (`:976,1111,940`). The writable bind is the queue lane but also a write surface to the whole checkout path.
- **MEDIUM — `docker-job-loop.sh` never surfaces child exit codes to the supervisor.** `while true; do run_job_once; sleep; done` (`:141-144`) swallows failures; a perpetually-failing job stays "running"/`fail` only in its JSON, and `restart: unless-stopped` never trips because the loop process itself never exits. Detection relies entirely on `docker-health.sh` parsing the status file.
- **MEDIUM — status-file key drift (`job_name`/`exit_code` vs `job`/`returncode`).** `docker-job-loop.sh:82,84` vs `docker-health.sh:250,252`. Works only via `or` fallback; brittle.
- **LOW — `compose()` is docker-only** (`bin/arclink-docker.sh:123`) while `bin/common.sh:run_compose` is runtime-agnostic; a podman-only host cannot run the main stack via `arclink-docker.sh` even though the Nextcloud sidecar can. `require_docker_compose` (`:185-190`) fails closed with a clear message.
- **LOW — placeholder-secret warning is non-blocking.** `docker-health.sh:474-476` only `warn`s if `change-me` Nextcloud/Postgres secrets are live; it does not fail the health check.
